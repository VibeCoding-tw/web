Module 1: Intuitive Lab

同步 vs 非同步：為什麼畫面卡住了？

SYNC

🚫 同步 (Synchronous)

程式碼一行一行執行。當遇到耗時操作（如連線）時，整個網頁的主執行緒會被「阻塞 (Block)」，導致按鈕無法點擊、動畫停止，使用者體驗極差。

ASYNC

⚡ 非同步 (Asynchronous)

使用 Promise 把耗時任務丟給瀏覽器背景處理。主執行緒繼續回應使用者操作。當任務完成時，再透過回呼或 await 繼續執行後續動作。

Async/Await：讓代碼像詩一樣優雅

1 告別 Callback Hell，擁抱語法糖

The Old Way (Callback Hell) Painful 😫 
navigator.bluetooth.requestDevice({...})
  .then(device => {
    return device.gatt.connect()
      .then(server => {
        return server.getPrimaryService(...)
          .then(service => {
             // ... 波動拳 👊
             return service.getCharacteristic(...)
          })
      })
  })
  .catch(error => { ... });
Async / Await Elegant ✨ 
async function connectBLE() {
  try {
    const device = await navigator.bluetooth.requestDevice(...);
    const server = await device.gatt.connect();
    const service = await server.getPrimaryService(...);
    const char = await service.getCharacteristic(...);
    
    console.log('Ready!');
  } catch (error) {
    console.error('Oops', error);
  }
}

2 Web BLE 連線鏈

Web BLE 的連線過程是嚴格的「階層式」操作。每一層都必須等待上一層 Resolved 才能繼續。

錯誤捕獲：Try...Catch

藍牙連線變數極多（距離遠、被拒絕、斷線）。使用 try...catch 包裹整個 async 函式，是保護 App 不會崩潰的唯一防線。

Step 1

requestDevice()

彈出視窗，等待使用者選擇設備。

Step 2

gatt.connect()

建立低功耗藍牙連線。這是最容易失敗的一步（Timeout, Error 133 等）。

Step 3

getPrimaryService()

尋找該設備提供的特定服務（Service UUID）。

Target

getCharacteristic()

取得最終的讀寫控制點 (Characteristic)。

作業 2-3

45 min

不同步超時保護 (Timeout)

硬體有時會死當，導致 await connect() 永久卡住。請實作「10 秒超時」機制。

Promise.race([connect(), timeout(10000)])

Impact

主動中斷連線嘗試，並提示使用者「連線逾時」，而不是讓網頁轉圈圈。

作業指引 (Assignment Guide)

作業 2-1：「逐步點亮」狀態機實作

  • 角色扮演： 系統架構師。
  • 執行方式： 學生需實作一個 UI 狀態標籤（如 <span id="status">），隨著 await 的每一階段（Request -> Connect -> Service -> Characteristic）更新內容。
  • 評分重點： 按鈕是否在 Connecting 狀態下被 disabled 以防止重複請求。

作業 2-2：處理並發與競爭 (Race Condition)

  • 角色扮演： 同步控制專家。
  • 執行方式： 當使用者快速點擊多個指令時，設計一個 isBusy flag 鎖定 GATT 操作。
  • 評分重點： 在快速點擊下是否還會出現 "GATT operation already in progress" 的錯誤。

作業 2-3：不同步超時保護 (Timeout)

  • 角色扮演： 穩定性測試工程師。
  • 執行方式： 使用 Promise.race 實作 10 秒超時機制。
  • 評分重點： 當硬體關閉時，網頁是否能在 10 秒後主動報錯並提示使用者，而非永久轉圈。
