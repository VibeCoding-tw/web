Chart.js 與 Canvas
即時資料視覺化

將感測器數據轉化為動態圖表：掌握 Canvas 渲染原理與 Sliding Window 數據緩衝技術。

🖌️ 1. 渲染原理：畫布 vs. 物件 (25 min)

  • ● Canvas API (即時畫布)：理解像素操作。像是一張白紙，你必須手動下指令「移動到 A 點，畫線到 B 點」。適合極高性能、超大量數據點。
  • ● Chart.js (資料視覺化庫)：基於 Canvas 封裝。你只需要提供數據陣列，它會幫你處理座標軸、縮放與動畫。適合快速建立專業儀表板。

🧩 學習任務

比較「靜態圖表」與「即時圖表」的差異。為什麼即時圖表需要不斷「抹除舊畫面」再「畫上新畫面」？

🔄 2. 資料結構：滑動窗口 (Sliding Window) (25 min)

// 滑動窗口邏輯實作
function updateBuffer(newValue) {
dataQueue.push(newValue);
if (dataQueue.length > 50) {
dataQueue.shift(); // 移除最舊數據
}
chart.update('none'); // 即時刷新
}

記憶體與效能管理

  • 佇列 (Queue) 概念： 使用 `push()` 增加新數據，並使用 `shift()` 移除最舊的數據。
  • 容量控制： 保持畫面上永遠只顯示最近的 50 或 100 個點，防止瀏覽器當掉。

🔍 學習任務：效能觀察

觀察當數據點過多（例如顯示 1000 點）時，圖表更新速度變慢的現象。

⚡ 3. 更新頻率與視覺感知 (10 min)

RequestAnimationFrame：理解為什麼要配合螢幕刷新率（通常是 60Hz）來繪圖，而不是數據一進來就無限制地刷新。

滾動窗口 (Rolling Window)：學習如何使用 Chart.js 建立動態更新的座標軸，並關閉動畫以釋放 CPU。

🧠 思考題

如果馬達回傳數據頻率是 1000Hz，我們真的需要每一筆都畫出來嗎？人眼能感知到的極限在哪？

作業指引 (Assignment Guide)

進階 7 - 即時繪圖原理

2-1 滑動視窗 (Sliding Window) 實作

角色扮演： 嵌入式開發者。

執行步驟：

  1. 建立一個固定長度為 50 的 JS 陣列。
  2. 每當從 BLE 接收到馬達 PWM 資料時，使用 push() 塞入新值並檢查長度。
  3. 若超過 50，則使用 shift() 剔除舊值。

評分重點： 資料緩衝區是否能穩定維持在固定長度，避免瀏覽器因數據無限制增長而崩潰。

2-2 多軸同步監控儀 (Multi-axis Chart)

角色扮演： 控制工程師。

執行步驟：

  • 在同一個 Chart.js 畫布上配置兩個 datasets (目標速度 vs. 實際速度)。
  • 關閉 Chart.js 的預設動畫 (Animation: false) 以換取極致更新效能。
  • 觀察 PID 控制過程中的誤差收斂圖形。

評分重點： 能否關閉動畫以換取效能，並觀察 PID 控制過程中的誤差收斂圖形。

2-3 Canvas 原生繪圖挑戰

角色扮演： 高性能運算專家。

執行步驟：

  • 棄用 Chart.js，改用原生 Canvas API (context.lineTo) 繪製一組每 10ms 更新一次的高頻波形圖。
  • 對於像素座標計算的理解（例如將資料值 0-255 映射到畫布高度 0-300）。
  • 理解繪圖頻率與螢幕刷新率的匹配。

評分重點： 對於像素座標計算的理解，以及繪圖頻率與螢幕刷新率的匹配。
