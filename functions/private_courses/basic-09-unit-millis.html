基礎 9 - 單元 1

擺脫 delay()：
millis() 與非阻塞邏輯

解鎖 ESP32 的多工潛力，讓你的機器人不再因為「等待」而錯失指令。

😈 1. 為什麼 delay() 是邪惡的？ (20 min)

阻塞 (Blocking)：當執行 delay(1000) 時，處理器會完全停下來什麼都不做。這就像你在等紅綠燈時閉上眼睛，完全不理會周遭發生的事。

指令被無視

如果這時無人車快撞牆了，你急忙按下「停止」鍵，但處理器還在 delay 中，你的指令將會被完全無視。

通訊中斷

對於藍牙或 Wi-Fi 連線，長時間的阻塞可能導致緩衝區溢位或連線超時，讓遙控器顯得反應遲鈍甚至斷線。

🎯 學習目標

理解「多工 (Multitasking)」的必要性。讓車子在閃爍 LED 的同時，還能以毫秒等級的速度檢查感測器與藍牙指令。

⏱️ 2. millis() 的本質：碼錶邏輯 (40 min)

核心概念：millis() 就像一個永不停止的碼錶，從電源開啟那一刻起紀錄經過的毫秒數。

🔄 非阻塞邏輯公式

不要叫電腦「等待 1 秒」，而是要它「檢查現在時間，減去上次執行的時間，是否大於 1 秒」。

$$\text{Current} - \text{Previous} \ge \text{Interval}$$

💎 關鍵技術：unsigned long

millis() 的數值會隨時間變得非常大（約 50 天才會溢位），因此必須使用 unsigned long 變數來存儲時間戳記。

// millis() Pattern
unsigned long lastTime = 0;
const long interval = 1000;

void loop() {
  unsigned long currentTime = millis();

  if (currentTime - lastTime >= interval) {
    lastTime = currentTime;
    // Do something...
  }
  // Loop continues instantly!
}

🤝 3. 合作式多工 (Cooperative Multitasking)

什麼是合作？每個任務都只佔用處理器極短的時間（檢查一下時間，如果不該做就馬上跳過），這讓多個任務看起來像是同時運行的。

為什麼發送數據會讓遙控失靈？

如果在發送遙測數據（如電量）時使用 delay()，處理器會卡在發送任務中，無法即時讀取新的遙控指令，造成嚴重的操作延遲。

🎯 線上任務：虛擬碼挑戰

嘗試撰寫一段邏輯結構，實現「LED 每 500ms 閃爍一次，同時每 100ms 檢查一次按鈕狀態」，且完全不使用 delay()。

IF (now - T1 > 500) { toggleLED(); T1 = now; }
IF (now - T2 > 100) { checkButton(); T2 = now; }
