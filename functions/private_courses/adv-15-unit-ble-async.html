Web Bluetooth 非同步與連線優化

深入探討 async/await 在硬體控制中的應用，打造具備「自動重補路、異常防湧」的健壯連線架構。

⚡

為什麼要用「非同步」？

CPU 運算速度比藍牙快數百萬倍。如果使用同步代碼，等待 ESP32 回應時整個 UI 會凍結。理解 Pending、Resolved、Rejected 是掌握連線邏輯的基礎。

🛡️

異常處理：當硬體不理你時

在現實世界中，使用者會取消選取、裝置會超出範圍。學習利用 Try...Catch 捕捉錯誤，並監聽 gattserverdisconnected 事件來實作自動重連。

作業指引 (Assignment Guide)

作業 2-1：線性化連線序列

  • 角色扮演： 非同步程式設計師。
  • 執行方式： 利用 async/await 重構繁瑣的 Promise Chain，確保連線、發現服務、開啟通知的步驟互不干擾。
  • 評分重點： 能否使用 try...catch 正確處理「使用者取消連線」與「逾時」異常，並在 UI 上給予明確提示。

作業 2-2：斷線自動重連機制

  • 角色扮演： 系統穩定性專家。
  • 執行方式： 監聽 gattserverdisconnected 事件，實作指數退避 (Exponential Backoff) 的重連邏輯。
  • 評分重點： 在突然關閉 ESP32 電源後再開啟時，網頁能否在 3 次重試內自動找回裝置並恢復控制。

作業 2-3：非同步並行處理：讀寫同步

  • 角色扮演： 通訊優化師。
  • 執行方式： 實現指令發送與遙測數據（Notify）並行且互不阻塞。
  • 評分重點： 檢查是否存在競爭條件 (Race Condition)，確保在高頻發送指令時，即時姿態回傳依然能保持穩定 FPS。
