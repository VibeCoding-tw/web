進階 14 - 單元 3

代碼重構與模組化設計

從「義大利麵代碼」進化到優雅的軟體架構。學習如何償還技術債，設計高內聚、低耦合的系統。

為什麼要重構？

為了趕進度而寫的快餐代碼是「技術債 (Technical Debt)」，利息就是未來修改時的高昂代價。重構不是增加新功能，而是讓代碼「更健康、更易讀、更好改」。

模組化原則

  • ✓高內聚 (High Cohesion)：一個檔案只做一件事
  • ✓低耦合 (Low Coupling)：模組間依賴越少越好
  • ✓關注點分離 (SoC)：感知、決策、執行分家

1 識別「代碼壞味道」 (Code Smells)

像聞牛奶一樣，有些代碼一打開就有「味道」。學會識別這些徵兆，是重構的第一步。

🐢 冗長函式 (Long Method)

一個 loop() 裡面寫了 500 行，包含連線、影像處理、馬達控制。看都看不完，更別說改了。

🔁 重複代碼 (Duplication)

違反 DRY (Don't Repeat Yourself) 原則。同樣的 JSON 解析邏輯複製貼上在三個地方，改了一個忘了改另外兩個。

🧱 硬編碼 (Hardcoding)

IP 地址、PID 參數直接寫死在程式碼深處。每次要微調參數都要重新編譯燒錄，浪費生命。

2 系統分層架構 (Separation of Concerns)

將你的無人車拆解為三個獨立的大腦區域，就像神經系統一樣。

👁️ 感知層 (Perception)

負責「看」與「聽」。影像讀取、感測器濾波、特徵提取。

Camera, IMU, LiDAR

🧠 決策層 (Logic)

負責「想」。狀態機切換 (FSM)、PID 運算、路徑規劃。

State Machine, PID

⚡ 執行層 (Action)

負責「做」。馬達 PWM 輸出、LED 顯示、蜂鳴器。

Motor, LED, Buzzer

作業指引 (Assignment Guide)

作業 3-1：「手術台」：解構大雜燴程式碼

  • 角色扮演： 代碼外科醫生。
  • 執行方式： 將提供一段所有功能都塞在 loop() 裡的「義大利麵代碼」，將其拆解為具備單一職責的 Function。
  • 評分重點： 代碼的可讀性（命名是否優雅）、全域變數的減少程度，以及是否成功將 Kp/Ki/Kd 封裝成 Config 物件。

作業 3-2：打造「通訊模組」Class

  • 角色扮演： 軟體架構師。
  • 執行方式： 利用 JavaScript Class 封裝 Web Bluetooth API，對外僅暴露 connect() 與 sendJson() 介面。
  • 評分重點： 介面隔離 (Interface Segregation) 的落實程度，以及主程式是否能完全不知曉底層 BLE 的實作細節。

作業 3-3：「插件化」影像處理鏈

  • 角色扮演： 系統擴充工程師。
  • 執行方式： 設計一個處理鏈模式 (Pipeline Pattern)，能夠動態地將「灰階」、「二值化」、「濾波」等模組 push 進入處理陣列中。
  • 評分重點： 是否落實「對擴展開放，對修改封閉」原則，在不修改主迴圈的情況下即可增加新濾鏡。
