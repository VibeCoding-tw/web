基礎 7 - 網路通訊架構

Wi-Fi 雙模式解析：
AP (自立門戶) vs. STA (加入大家庭)

掌握 ESP32 的通訊雙重身分，打造能在任何環境下連線的產品級智慧系統。

🗼 1. AP (Access Point) 模式：自立門戶 (20 min)

核心概念：ESP32 自己當「基地台」。就像車子背著一台小迷你路由器，手機可以直接連到車子發出的 Wi-Fi。

💪 優點

  • • 不依賴外部網路。
  • • 去到哪裡都能連線（適合戶外賽道）。
  • • 設備間可形成獨立的私有網段。

⚠️ 缺點

  • • 手機連上後通常無法上網。
  • • 連線距離受限於 ESP32 發射功率。
  • • 同時連進來的人數有物理限制。

🧐 學習任務

查閱 SSID 與 Password 的設定規範。思考：為什麼 AP 模式通常會有「最大連線人數」限制？這與 ESP32 的記憶體 (RAM) 有什麼關係？

🏠 2. STA (Station) 模式：加入大家庭 (20 min)

核心概念：ESP32 像手機一樣，「加入」現有的路由器（Router）。

💪 優點

  • • 可透過路由器進行長距離遠端控制。
  • • 車子可同時連網（下載更新或傳數據）。
  • • 穩定的網路基礎設施。

⚠️ 缺點

  • • 需要環境中有路由器。
  • • 路由器負載太重時延遲會變高。
  • • 依賴密碼正確與路由器開啟。

⚙️ 學習任務：DHCP 協定

為什麼車子加入網路後，IP 地址會變動？理解如何設定「固定 IP (Static IP)」，這就像幫無人車在社區裡租一個固定的停車位，確保控制端永遠找得到它。

⚡ 3. AP + STA (Dual Mode)：最強混合體 (20 min)

核心概念：同時開啟兩種模式，讓設備具備高度的彈性。

🛒 經典應用場景：配網流程 (Provisioning)

你買了一台新的智慧插座或掃地機器人，它是如何學會你家 Wi-Fi 密碼的？

  1. 插座啟動 AP 模式，發出名稱為 "Smart_XXX" 的 Wi-Fi。
  2. 你的手機連線到這個 AP，進入設定網頁。
  3. 你在手機上輸入你家的 Wi-Fi SSID 與密碼。
  4. 插座收到後，啟動 STA 模式 把身分切換到你家路由器。

作業指引 (Assignment Guide)

【作業 1-1】我的領地：AP 模式與強制門戶

  • 角色扮演： 區域網路管理員 (Local Network Administrator)。
  • 執行方式： 實作自定義 SSID (MyRobot_01) 與網段 (192.168.4.1)。測試連線穩定性，觀察多台手機連入時是否會互相干擾 Web 控制介面。並思考如何讓手機連上後自動彈出控制網頁。
  • 評分重點 : 成功建立穩定 AP，手機可正常加入並透過瀏覽器開啟指定 IP。

【作業 1-2】智慧尋路：STA 模式與斷線重連

  • 角色扮演： 穩定性測試工程師 (Stability Test Engineer)。
  • 執行方式： 撰寫程式碼掃描現場 Wi-Fi 並加入。實作 Wi-Fi 事件監聽 (Event Handler)，模擬路由器斷電場景，測試車子是否能在電力恢復後 30 秒內「自動重連」。
  • 評分重點 : 重連邏輯健全（非阻塞式），具備逾時處理與狀態日誌輸出。

【作業 1-3】終極備援：自動切換模式

  • 角色扮演： 強韌系統架構師 (Resilient System Architect)。
  • 執行方式： 實作 Fallback 邏輯：啟動後先嘗試連接預設 STA (5秒)，若失敗則自動轉為 AP 模式。利用內建 LED 燈號區分模式（如：STA 模式長亮，AP 模式緩慢閃爍）。
  • 評分重點 : 模式切換流暢，無系統死機現象，且具備清楚的視覺反饋。

GitHub Classroom 詳細操作指南

第一步：準備範本倉庫 (Template Repo)

  1. 在 GitHub 建立新 Repo（名為 basic-07-wifi-modes-template）。
  2. 上傳包含 `WiFi.h` 引用但「未實作模式切換邏輯」的 C++ 專案框架。
  3. 在 Repo 設定中勾選 "Template repository"。

第二步：建立 Classroom 作業

  1. 進入 GitHub Classroom 建立「Individual Assignment」。
  2. 連結上述的範本倉庫，並勾選 "Enable feedback pull requests"。

第三步：學生端操作 (Flipped Workflow)

  1. 學生領取作業，獲得個人專屬 Repo。
  2. 實作 AP 高強度發射功率設定、STA 事件監聽重連與 Fallback 自動備援機制。
  3. 於不同網路環境（教室 WiFi vs 手機熱點）驗證通過後 Push 代碼 提交進度。

第四步：教師回饋 (Review)

  1. 進入 Feedback PR 的 Files Changed。
  2. 針對學生的「重連指數策略 (Backoff)」與「事件處理結構」予以回饋。

main.cpp (Event-based Reconnect)

WiFi.onEvent([](WiFiEvent_t event, WiFiEventInfo_t info){
  Serial.print("WiFi Lost! Reason: ");
  Serial.println(info.wifi_sta_disconnected.reason);
  WiFi.begin(ssid, password);
}, ARDUINO_EVENT_WIFI_STA_DISCONNECTED);

C. 自動評分 (Autograding) 的設計

  • 模式宣告檢測：： 檢查程式碼中是否包含 WIFI_AP_STA 同時開啟或切換的 API 調用。
  • 狀態日誌匹配：： 驗證學生提交的 README 中是否能正確羅列出環境中至少 3 個 WiFi 訊號的 RSSI。
