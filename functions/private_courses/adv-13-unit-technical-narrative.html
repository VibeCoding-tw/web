進階 13 - 單元 3

視覺敘事：如何展示技術專案

技術再強，沒人看懂也是白搭。學習用「故事」和「視覺」為你的程式碼注入靈魂，打造令人驚艷的作品集。

為什麼要學技術敘事？

在開源社群或求職面試中，Reviewer 只有 30 秒能被你吸引。單純貼 GitHub code 並不足夠，你需要一個引人入勝的故事 (Hero's Journey) 和直觀的視覺展示，讓不懂代碼的人也能感受到技術的價值。

本單元目標

  • ✓掌握「英雄之旅」技術敘事框架
  • ✓設計「一眼就懂」的系統架構圖
  • ✓製作專業質感的 Demo 影片

1 英雄之旅：技術敘事的框架

別只寫 "I made a car."，試著講一個故事。好的技術敘事遵循「英雄之旅」的結構，讓讀者跟著你一起經歷開發的過程。

核心結構 (Structure)

  • 挑戰 (The Problem)：
    為什麼要做這台車？解決了什麼痛點（如：狹窄空間的物流、危險環境探勘）？
  • 掙扎 (The Struggle)：
    開發中遇到了什麼技術障礙（如：搖桿數據延遲 - causing crash、CORS 錯誤 - blocked data）？不要害怕暴露弱點，這才是真實的工程。

解決與成果 (Result)

  • 突破 (The Solution)：
    你用了什麼黑科技解決它（如：引入 FSM 穩定狀態、數據歸一化優化控制手感）？這是展示技術深度的時刻。
  • 成果 (The Result)：
    最終的性能表現。用數據說話（如：延遲從 500ms 降至 50ms、辨識準確率達 95%）。

2 視覺化工具：讓邏輯「被看見」

看不見的代碼很難被欣賞。我們需要將抽象的數據流 (Data Flow) 和系統架構 (Architecture) 具象化。

動態圖表 (Dynamic Charts)

使用 Chart.js 等函式庫將感測器數據轉化為動態曲線。

  • 即時馬達轉速 (RPM)
  • 電池電壓監控
  • PID 誤差修正曲線

架構圖設計 (Architecture Diagrams)

清楚展示前端、藍牙、硬體三方的關係。工具推薦：Lucidchart, Figma, Excalidraw

  • 分層架構 (Layered Architecture)
  • 數據流向 (Data Flow)

3 錄影與剪輯技巧

沒有影片就沒真相 (Pics or it didn't happen)。一個好的 Demo 影片勝過千言萬語。

🎥
POV 第一人稱視角

把手機架在頭上或胸前，展示「操作者」的真實體驗，讓觀眾有代入感。

🖼️
PiP 子母畫面

左邊放手機螢幕錄影，右邊放車子實際跑動畫面。同步展示「操作 -> 反應」。

🎬
敘事節奏

問題定義 (5s) -> 技術特寫 (10s) -> 實際功能演示 (15s) -> 未來展望

📝 實作任務 (Classroom Challenges)

Task 3-1 「一眼就懂」的架構圖設計 (60 min)

任務說明 (Mission)

利用圖形工具，將你無人車的「數據流」畫出來。目標是製作一張不需要解釋，別人就能看懂系統運作邏輯的資訊圖表。

  • 分層呈現：清楚區分 UI 層 (Web/App)、通訊層 (BLE/WiFi) 與硬體執行層 (ESP32/Motor)。
  • 視覺指引：用箭頭標示數據（如 Joystick $(x,y)$）包含數學轉換公式，最終變成馬達 PWM 的過程。

Task 3-2 動態數據儀表板 (60 min)

任務說明 (Mission)

在你的控制網頁上加入實時的「性能監控器」。目標是在演示時，除了看車子在跑，還能看到背後的「數據心跳」。

  • 波形繪製：利用 Canvas 或 Chart.js 繪製馬達輸出 (PWM) 或 PID 誤差的即時波形。
  • 數值儀表：實作圓形進度條或數字面板，顯示當前的 CPU 負載 (Loop Time) 或電池電壓。

Task 3-3 30 秒電梯簡報 (Elevator Pitch) (45 min)

任務說明 (Mission)

錄製一段現場展示影片，具備專業質感的技術成果短片。

  • 實機演示：邊操作邊說明。例如：「現在切換到自律模式，你可以看到畫面上搖桿變暗了...」。
  • 性能特寫：特寫展示技術細節，如「死區 (Deadzone)」消除後的微動反應，或「指數曲線 (Expo)」帶來的平滑起步效果。
