進階 13 - 單元 3

視覺敘事：如何展示技術專案

技術再強，沒人看懂也是白搭。學習用「故事」和「視覺」為你的程式碼注入靈魂，打造令人驚艷的作品集。

為什麼要學技術敘事？

在開源社群或求職面試中，Reviewer 只有 30 秒能被你吸引。單純貼 GitHub code 並不足夠，你需要一個引人入勝的故事 (Hero's Journey) 和直觀的視覺展示，讓不懂代碼的人也能感受到技術的價值。

本單元目標

  • ✓掌握「英雄之旅」技術敘事框架
  • ✓設計「一眼就懂」的系統架構圖
  • ✓製作專業質感的 Demo 影片

1 英雄之旅：技術敘事的框架

別只寫 "I made a car."，試著講一個故事。好的技術敘事遵循「英雄之旅」的結構，讓讀者跟著你一起經歷開發的過程。

核心結構 (Structure)

  • 挑戰 (The Problem)：
    為什麼要做這台車？解決了什麼痛點（如：狹窄空間的物流、危險環境探勘）？
  • 掙扎 (The Struggle)：
    開發中遇到了什麼技術障礙（如：搖桿數據延遲 - causing crash、CORS 錯誤 - blocked data）？不要害怕暴露弱點，這才是真實的工程。

解決與成果 (Result)

  • 突破 (The Solution)：
    你用了什麼黑科技解決它（如：引入 FSM 穩定狀態、數據歸一化優化控制手感）？這是展示技術深度的時刻。
  • 成果 (The Result)：
    最終的性能表現。用數據說話（如：延遲從 500ms 降至 50ms、辨識準確率達 95%）。

2 視覺化工具：讓邏輯「被看見」

看不見的代碼很難被欣賞。我們需要將抽象的數據流 (Data Flow) 和系統架構 (Architecture) 具象化。

動態圖表 (Dynamic Charts)

使用 Chart.js 等函式庫將感測器數據轉化為動態曲線。

  • 即時馬達轉速 (RPM)
  • 電池電壓監控
  • PID 誤差修正曲線

架構圖設計 (Architecture Diagrams)

清楚展示前端、藍牙、硬體三方的關係。工具推薦：Lucidchart, Figma, Excalidraw

  • 分層架構 (Layered Architecture)
  • 數據流向 (Data Flow)

3 錄影與剪輯技巧

沒有影片就沒真相 (Pics or it didn't happen)。一個好的 Demo 影片勝過千言萬語。

🎥
POV 第一人稱視角

把手機架在頭上或胸前，展示「操作者」的真實體驗，讓觀眾有代入感。

🖼️
PiP 子母畫面

左邊放手機螢幕錄影，右邊放車子實際跑動畫面。同步展示「操作 -> 反應」。

🎬
敘事節奏

問題定義 (5s) -> 技術特寫 (10s) -> 實際功能演示 (15s) -> 未來展望

實作任務：說一個動人的技術故事

打造你的專屬作品集，讓冷冰冰的代碼散發溫度。

作業指引 (Assignment Guide)

作業 3-1：「架構地圖」：系統方塊圖繪製

  • 角色扮演： 技術文件作家。
  • 執行方式： 使用 Mermaid 或網頁工具繪製系統架構圖，包含前端、BLE 與 ESP32 的數據流向。
  • 評分重點： 能否清晰區分出「輸入端」（感測器）、「處理端」（PID）與「輸出端」（馬達），並正確標註通訊協議。

作業 3-2：「一分鐘專題」：技術演示影片

  • 角色扮演： 產品經理 (PM)。
  • 執行方式： 錄製 60 秒的展示影片，包含循進、過彎與遇到異常（斷訊）時的自動停機演示。
  • 評分重點： 技術口條是否清晰，演示過程是否能突顯系統的「穩定性」優勢。

作業 3-3：「開源文件」：README 與 Wiki

  • 角色扮演： 開源社群維護者。
  • 執行方式： 在 GitHub 上完善 README.md，包含安裝步驟、功能介紹與代碼導讀。
  • 評分重點： 文件內容是否能讓一個「完全沒碰過這專案的人」在 10 分鐘內成功部署並運行車子。

🚀 在 GitHub Classroom 的實戰部署建議

A. 範本倉庫 (Template Repo) 結構 

.
├── README.md               # 待學生填充的專案展示首頁
├── docs/
│   ├── architecture.png    # 系統方塊圖與數據流向
│   └── manual.md           # 硬體連接與軟體部署手冊
└── assets/                 # 技術展示影片與系統運行 GIF

B. 評估 (Evaluation) 與同儕審查

📝

技術文件完整度： 掃描 README，確認包含功能列表、架構說明與性能指標。

🤝

同儕回饋機制： 讓學員互相審查，指出「最難懂的部分」並給予建議。

🛠️ GitHub Classroom 詳細操作指南

STEP 1

準備範本專案

上傳 README 專業範本。預先設定 GitHub Pages 路徑規範。

STEP 2

建立 Classroom 作業

設定「互評 (Peer Review)」機制。討論技術與產品的平衡點。

STEP 3

學生端 (Flipped)

執行架構繪圖與展示影片錄製。提交作品集 README PR。執行 Git Push。

STEP 4

教師回饋 (Review)

針對技術溝通能力給予評價，審核專案展示的專業性與精確度。

🔄 翻轉教室的實施策略

階段 學生行動 (Git Workflow) 教師/平台角色
課前 整理實驗數據，篩選最具挑戰性的成果作為作品集核心。 檢查 Dashboard 進度，提供「技術敘事：如何介紹你的硬體解方」導讀。
課中 領取 Classroom 作業，繪製系統方塊圖並錄製展示影片初稿。 即時連線排除阻礙，引導進行「閃電互評」，檢核 README 可行性。
課後 修正文件並發布 Portfolio 並 Push，閱讀對技術敘事能力的 PR 回饋。 核發評量，引導學員思考技術品牌在職涯中的價值。
