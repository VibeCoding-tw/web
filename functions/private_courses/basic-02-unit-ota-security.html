Security & Integrity

守護你的硬體防線：防止非法篡改與錯誤更新。
學習如何建立數位守門員，確保韌體更新的安全與穩定。

⚠️ 為什麼 OTA 需要安全機制？ (20 min)

沒有保護的 OTA，就像是家門大開的保險箱。

攔截與偽造

理解中間人攻擊 (MITM)。若無保護，駭客可上傳惡意二進制檔接管無人車，甚至造成危險。

版本倒退風險

若將舊版的 Bug 韌體重新刷入新版機器，可能導致功能失效或硬體損壞（Anti-rollback）。

硬體不匹配

若將 ESP32-S3 的韌體刷入 ESP32-C3，系統將無法啟動。必須在更新前檢查硬體 ID。

學習任務：思考與討論

思考生活中常見的軟體更新（如手機系統），更新前系統會檢查哪些資訊？這對用戶的安全性有何保障？

🔐 密碼保護：HTTP 基本驗證 (20 min)

HTTP Basic Auth

利用瀏覽器內建機制，在請求更新頁面前彈出使用者名稱與密碼視窗。這是最簡單有效的防禦手段。

伺服器端驗證流程

  • 瀏覽器發送帶有 Auth Header 的請求
  • ESP32 攔截請求並提取帳密
  • 比對儲存在 NVS (Preferences) 中的雜湊值
  • 通過則回傳網頁，失敗則回傳 401
esp32-auth.cpp

驗證代碼片段

server.on("/update", HTTP_GET, []() {
  if (!server.authenticate(www_username, www_password)) {
    return server.requestAuthentication();
  }
  server.send(200, "text/html", updateIndex);
});
學習任務： 查閱 ESP32 WebServer 庫中的 authenticate() 函數用法，思考如何實現「動態密碼」。

🏷️ 版本檢查與中繼資料 (20 min)

1. 語義化版本

理解 MAJOR.MINOR.PATCH (如 v1.2.3) 的定義。主要更新、次要更新、修復補丁。

2. 韌體標頭 (Header)

學習如何自定義結構。在編譯時將「版本號」與「目標型號」嵌入 .bin 檔的前段位置。

3. 更新邏輯判斷

New > Current? Target == Current? 只有在條件滿足時，才調用 Update.begin()。

如何防止別人惡意更新？

除了密碼，更深層的防護是「簽章驗證」。在本課程中，我們先實作基礎的版本與硬體比對，確保系統內容的正確性。

關鍵概念：防呆機制

當上傳一個不正確的 .bin 檔時，ESP32 應該在寫入 Flash 之前就發現錯誤（讀取 Header），而不是寫到一半出錯。

作業指引 (Assignment Guide)

作業 3-1：「數位守門員」實作 OTA 密碼鎖

  • 角色扮演： 安全工程師 (Security Engineer)。
  • 執行方式： 為 /update 路由增加 HTTP Basic Auth。帳密應從 NVS (Preferences) 讀取，並實作「密碼錯誤 3 次後鎖定 5 分鐘」的簡單計數器。
  • 評分重點： 帳密驗證邏輯的正確性，以及密碼錯誤時的鎖定機制是否生效。

作業 3-2：「防呆機制」韌體版本號檢查

  • 角色扮演： 韌體驗證工程師 (Firmware QA)。
  • 執行方式： 定義 APP_VERSION，在上傳前讀取 .bin 標記。若版本較舊，利用 Update.abort() 中止更新並回傳「禁止降級」。
  • 評分重點： 能在 Update.write() 之前攔截錯誤版本，並提供友善的報錯訊息。

作業 3-3：「硬體適配」硬體 ID 驗證

  • 角色扮演： 系統整合專家 (System Integrator)。
  • 執行方式： 使用 ESP.getChipModel() 檢測硬體型號。在 OTA 封包中加入應配對的硬體名稱。嘗試將錯誤型號韌體上傳並觀察攔截結果。
  • 評分重點： 確保韌體與硬體型號匹配，防止因硬體架構不同而導致的更新後無法開機。
