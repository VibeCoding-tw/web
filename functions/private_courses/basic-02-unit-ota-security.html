OTA 安全性：密碼與版本檢查
Security & Integrity

守護你的硬體防線：防止非法篡改與錯誤更新。
學習如何建立數位守門員，確保韌體更新的安全與穩定。

⚠️ 為什麼 OTA 需要安全機制？ (20 min)

沒有保護的 OTA，就像是家門大開的保險箱。

攔截與偽造

理解中間人攻擊 (MITM)。若無保護，駭客可上傳惡意二進制檔接管無人車，甚至造成危險。

版本倒退風險

若將舊版的 Bug 韌體重新刷入新版機器，可能導致功能失效或硬體損壞（Anti-rollback）。

硬體不匹配

若將 ESP32-S3 的韌體刷入 ESP32-C3，系統將無法啟動。必須在更新前檢查硬體 ID。

學習任務：思考與討論

思考生活中常見的軟體更新（如手機系統），更新前系統會檢查哪些資訊？這對用戶的安全性有何保障？

🔐 密碼保護：HTTP 基本驗證 (20 min)

HTTP Basic Auth

利用瀏覽器內建機制，在請求更新頁面前彈出使用者名稱與密碼視窗。這是最簡單有效的防禦手段。

伺服器端驗證流程

  • 瀏覽器發送帶有 Auth Header 的請求
  • ESP32 攔截請求並提取帳密
  • 比對儲存在 NVS (Preferences) 中的雜湊值
  • 通過則回傳網頁，失敗則回傳 401
esp32-auth.cpp

驗證代碼片段

server.on("/update", HTTP_GET, []() {
  if (!server.authenticate(www_username, www_password)) {
    return server.requestAuthentication();
  }
  server.send(200, "text/html", updateIndex);
});
學習任務： 查閱 ESP32 WebServer 庫中的 authenticate() 函數用法，思考如何實現「動態密碼」。

🏷️ 版本檢查與中繼資料 (20 min)

1. 語義化版本

理解 MAJOR.MINOR.PATCH (如 v1.2.3) 的定義。主要更新、次要更新、修復補丁。

2. 韌體標頭 (Header)

學習如何自定義結構。在編譯時將「版本號」與「目標型號」嵌入 .bin 檔的前段位置。

3. 更新邏輯判斷

New > Current? Target == Current? 只有在條件滿足時，才調用 Update.begin()。

如何防止別人惡意更新？

除了密碼，更深層的防護是「簽章驗證」。在本課程中，我們先實作基礎的版本與硬體比對，確保系統內容的正確性。

關鍵概念：防呆機制

當上傳一個不正確的 .bin 檔時，ESP32 應該在寫入 Flash 之前就發現錯誤（讀取 Header），而不是寫到一半出錯。

// 預想的驗證邏輯

startUpdate();

} else { abort("Hardware mismatch!"); }

} else { abort("Rollback prohibited!"); }

課程教材 - 內部使用

作業指引 (Assignment Guide)

1. GitHub Classroom 設定

  • 點擊儀表板「領取作業」按鈕，獲取 `basic-02-ota-security` 作業儲存庫。
  • 檢視專案中的 `certs` 資料夾，了解根憑證 (Root CA) 的存放位置。
  • 確保 PlatformIO 專案已正確配置 `board_build.embed_txtfiles` 以嵌入憑證。

2. 翻轉課堂學習策略

  • 課前預習：觀看「物聯網安全：OTA 攻擊面分析」影片，理解為什麼明文升級很危險。
  • 課中實作：實施 HTTPS 升級實驗，並嘗試使用錯誤的簽章來誘發升級失敗。
  • 課後思考：如果 private key（私鑰）外洩，對於已售出的數萬台設備會有什麼影響？如何補救？

3. 單元實作任務

  • 作業 2-1 (40%)：建立 HTTPS 連線，並成功獲取伺服器上的韌體資訊檔。
  • 作業 2-2 (30%)：實作韌體簽章驗證 (Firmware Signature)，確保只有授權的二進位檔能被執行。
  • 作業 2-3 (30%)：撰寫安全分析報告，列出三種可能的 OTA 攻擊路徑與對應防治方法。

講師手冊 (Instructor Guide)

1. 教學重點 (Teaching Points)

  • 對比 HTTP vs HTTPS 在 OTA 上的效能與安全權衡。
  • 解釋「公鑰基礎設施 (PKI)」在嵌入式系統中的簡化應用。
  • 強調「防止降級攻擊 (Anti-rollback)」的必要性：防止駭客將系統退回有漏洞的舊版本。

2. 常見問題 (Common Issues)

  • SSL 握手失敗 (Handshake failed)：通常是 ESP32 系統時間不正確導致憑證過期判定，需先實作 NTP 校時。
  • 堆疊溢位 (Stack overflow)：HTTPS 的加密運算極耗記憶體，需注意任務的 Stack 大小分配。
  • 憑證管理：提醒學生切勿將私鑰上傳到公開的 GitHub Repo。

3. 評分標準 (Grading Criteria)

  • 基礎分：能正確載入根憑證並與 HTTPS 伺服器通訊。
  • 進階分：能理解安全啟動 (Secure Boot) 與加密 OTA 的關聯性。

B. 自動評估 (Autograding) 的設計

  • 安全協定檢測： 透過測試腳本搜尋代碼中的 WiFiClientSecure 使用情況，確保升級流程並非明文 HTTP。

GitHub Classroom 詳細操作指南

第一步：準備範本倉庫 (Template Repo)

建立包含基礎 OTA 外殼但「未實作 HTTPS 加密」的 Repo，於 GitHub 設定中勾選 "Template repository"。

第二步：建立 Classroom 作業

建立 Individual Assignment，連結範本並啟用 "Feedback pull requests" 以便對其憑證管理與 NTP 校核邏輯進行點評。

第三步：學生端操作 (Flipped Workflow)

學生領取作業，獲得個人專屬 Repo，實作雙向憑證校驗邏輯與簽章驗證功能後 Push。

第四步：教師回饋 (Review)

進入 Feedback PR，針對學生的「私鑰保護機制」與「SSL 資源優化」設計予以回饋。
