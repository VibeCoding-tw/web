進階 14 - 單元 1

軟體工程的除錯藝術

不要猜測，要證實。從「會解 Bug」進化到「會寫出不容易產生 Bug 的代碼」。

1 除錯的心理學與科學方法

除錯不僅僅是找錯誤，更是一場心理戰。很多時候，我們被自己的「假設」蒙蔽了雙眼。

🦆 橡皮鴨除錯法 (Rubber Duck Debugging)

向他人（或桌上的鴨子）逐行解釋你的程式碼。當你試圖用自然語言表達邏輯時，往往會在這過程中自己發現邏輯漏洞。

🔍 二分搜尋法定位 (Binary Search)

當程式有 1000 行時，如何快速縮小範圍？透過註解掉一半的程式碼，判斷問題還在不在。重複此步驟，只需幾次就能鎖定 Bug 位置。

科學方法核心：

不要猜測 (Don't Guess)。遵循「假設 —> 實驗 —> 觀察 —> 結論」的循環。回憶你上次解決 Bug，是靠運氣改對的，還是真的理解了原因？

2 除錯工具箱的深度應用

日誌的藝術 (Logging)

`console.log` 人人都會，但重點是「上下文資訊」。

  • 不僅印出誤差值，還要印出當時的系統時間。
  • 印出感測器狀態與變數型別 (typeof)。
  • 使用不同的 Log Level：INFO, WARN, ERROR。

斷點 (Breakpoints) & 堆疊追蹤

讓時空暫停的魔法。

  • 暫停時空：在特定行號停止執行，觀察變數的即時數值。
  • Stack Trace：看懂錯誤訊息的「犯罪現場」。依據堆疊判斷是你的代碼錯了，還是引用庫的問題？
預習任務：在瀏覽器 F12 > Sources 中，對一段迴圈設定斷點並逐步執行 (Step Over)。

3 硬體與網路除錯的特殊性

⚡ 非同步的陷阱 (Async Traps)

在藍牙或網路通訊中，Bug 往往具有隨機性 (Race Condition)。如果你的車子「偶爾」會轉向過度，除了程式碼，還要考慮：

電量不足 網路延遲 感測器雜訊 記憶體溢出

🛠️ 關鍵實踐

  • 1. 從 Serial Monitor 的錯誤訊息 (如 Guru Meditation Error) 定位 ESP32 的記憶體溢出位置。
  • 2. 使用 Try-Catch 捕獲異常，避免系統直接崩潰 (Crashing)。

📝 實作任務 (Classroom Challenges)

Task 1-1 「命案現場」：找出埋好的 Bug (60 min)

任務說明 (Mission)

教師提供一段看似正確但無法運作的無人車控制代碼，你需要找出隱藏的 3 個 Bug。

  • 邏輯型：變數名稱拼錯 (如 error 寫成 eror)。
  • 類型型：字串與數字相加 (JSON 讀取的 "50" 沒轉 Number 就運算)。
  • 非同步型：在藍牙尚未連線完成就發送指令。
🚫 限制：不得直接閱讀原始碼，必須先透過 console.log 或 斷點 觀察變數變化。撰寫「除錯日誌」記錄你的假設與實驗。

Task 1-2 「通訊黑盒」：網路封包抓包與分析 (60 min)

任務說明 (Mission)

當網頁無法控制車子時，判斷問題是在「網頁端」、「網路傳輸」還是「ESP32 端」。掌握跨裝置除錯的「隔離技術」。

  • Network 面板：檢查 HTTP 請求狀態 (200 OK vs 404/500)。
  • 模擬弱網：人為增加 500ms 延遲，觀察控制邏輯是否崩潰。
  • 封包比對：檢查發出的 JSON 格式是否與 ESP32 規格一致。

Task 1-3 「防禦性編程」：打造不壞的代碼 (45 min)

任務說明 (Mission)

重構你的代碼，增加「容錯」與「自我診斷」能力。

  • 輸入驗證：處理數據前加入 if (isNaN(value)) 判斷。
  • 逾時機制：若藍牙指令 1 秒內沒回應，自動執行 stop() 以策安全。
  • 錯誤回饋：建立「錯誤訊息視窗 / Toast」，發生錯誤時顯示具體原因，而非網頁空白。
