進階 14 - 單元 1

軟體工程的除錯藝術

不要猜測，要證實。從「會解 Bug」進化到「會寫出不容易產生 Bug 的代碼」。

1 除錯的心理學與科學方法

除錯不僅僅是找錯誤，更是一場心理戰。很多時候，我們被自己的「假設」蒙蔽了雙眼。

🦆 橡皮鴨除錯法 (Rubber Duck Debugging)

向他人（或桌上的鴨子）逐行解釋你的程式碼。當你試圖用自然語言表達邏輯時，往往會在這過程中自己發現邏輯漏洞。

🔍 二分搜尋法定位 (Binary Search)

當程式有 1000 行時，如何快速縮小範圍？透過註解掉一半的程式碼，判斷問題還在不在。重複此步驟，只需幾次就能鎖定 Bug 位置。

科學方法核心：

不要猜測 (Don't Guess)。遵循「假設 —> 實驗 —> 觀察 —> 結論」的循環。回憶你上次解決 Bug，是靠運氣改對的，還是真的理解了原因？

2 除錯工具箱的深度應用

日誌的藝術 (Logging)

`console.log` 人人都會，但重點是「上下文資訊」。

  • 不僅印出誤差值，還要印出當時的系統時間。
  • 印出感測器狀態與變數型別 (typeof)。
  • 使用不同的 Log Level：INFO, WARN, ERROR。

斷點 (Breakpoints) & 堆疊追蹤

讓時空暫停的魔法。

  • 暫停時空：在特定行號停止執行，觀察變數的即時數值。
  • Stack Trace：看懂錯誤訊息的「犯罪現場」。依據堆疊判斷是你的代碼錯了，還是引用庫的問題？
預習任務：在瀏覽器 F12 > Sources 中，對一段迴圈設定斷點並逐步執行 (Step Over)。

3 硬體與網路除錯的特殊性

⚡ 非同步的陷阱 (Async Traps)

在藍牙或網路通訊中，Bug 往往具有隨機性 (Race Condition)。如果你的車子「偶爾」會轉向過度，除了程式碼，還要考慮：

電量不足 網路延遲 感測器雜訊 記憶體溢出

🛠️ 關鍵實踐

  • 1. 從 Serial Monitor 的錯誤訊息 (如 Guru Meditation Error) 定位 ESP32 的記憶體溢出位置。
  • 2. 使用 Try-Catch 捕獲異常，避免系統直接崩潰 (Crashing)。

作業指引 (Assignment Guide)

作業 4-1：「命案現場」：找出預埋 Bug

  • 角色扮演： 代碼偵探。
  • 執行方式： 在不偷看正確代碼的情況下，利用 console.log() 與斷點找出代碼中隱藏的邏輯錯誤、類型錯誤與非同步陷阱。
  • 評分重點： 能否在「除錯日誌」中詳細紀錄：假設、實驗方法、觀測結果與最後的修正方案。

作業 4-2：「通訊黑盒」：網路封包抓包分析

  • 角色扮演： 運維工程師。
  • 執行方式： 當網頁無法控制車子時，判斷是「網頁端」、「網路延遲」還是「ESP32」的問題。
  • 評分重點： 正確使用 Chrome Network 面板截圖證明 HTTP/BLE 指令的發送狀態，並提出隔離測試的證據。

作業 4-3：「防禦性編程」：打造不壞的代碼

  • 角色扮演： 系統穩定性專家。
  • 執行方式： 重構代碼，增加輸入驗證（isNaN 檢查）與自動超時保護機制。
  • 評分重點： 代碼在極端異常輸入（如亂碼、NaN）下是否能維持運行且正確報錯（Toast 提示）。

🛠️ GitHub Classroom 詳細操作指南

STEP 1

準備範本專案

上傳預埋 Bug 代碼與除錯日誌模板。隨後 README 需標註系統異常現象。

STEP 2

建立 Classroom 作業

設定 PR 模板要求附上 F12 除鎖截圖。Discussion 建立「Bug 通緝榜」。

STEP 3

學生端 (Flipped)

領取作業，限時找出隱藏漏洞。提交包含假設、驗證與修正方案的日誌 PR。執行 Git Push。

STEP 4

教師回饋 (Review)

針對學員的「除錯邏輯推導」與「防禦性思考」評分，而不僅是修復 Bug 本身。

🔄 翻轉教室的實施策略

階段 學生行動 (Git Workflow) 教師/平台角色
課前 學習 F12 工具基礎，觀看「從觀測到假設：有系統找出異常」預習影片。 檢查 Dashboard 進度，提供「除錯的藝術：為什麼工程師 80% 時間都在修 Bug」導讀。
課中 領取 Classroom 作業，挑戰「命案現場」偵探任務，討論網路與硬體異常差異。 即時連線排除阻礙，分享正規觀測方法，啟發對「防禦性編程」的強烈需求。
課後 完成異常報告並 Push，閱讀回覆 PR 以最佳化代碼健壯性。 核發評量，引導思考系統穩定性對於關鍵任務軟體的決定性。
