基礎 9 - 單元 2

精通 硬體定時器
& ISR 中斷服務

超越 millis() 的極致精確度，讓處理器為你「插隊」執行關鍵任務。

⏰ 1. 什麼是硬體定時器？ (30 min)

核心概念：想像處理器內部有一個獨立的小鬧鐘。主程式（Loop）在跑的時候，鬧鐘時間一到，處理器會「強行暫停」手邊工作，去執行特定任務。

⚔️ 與 millis() 的差異

millis()：被動查詢

「看錶」，必須在 loop 裡面手動檢查時間，會受其他程式阻塞影響。

定時器中斷：主動通知

「鬧鐘響」，時間到直接插隊執行，不受 loop 阻塞影響，精度極高。

🎯 學習目標

理解 ESP32 擁有 4 個硬體定時器，可以達到微秒 (\(\mu s\)) 等級的精確度。

🚫 2. 中斷服務常式 (ISR) 的三大禁忌 (30 min)

關鍵概念：ISR 必須「快進快出」。

不可使用 delay()

會導致系統鎖死，因為中斷已經停止了時間流逝的計數環境。

不可進行耗時運算

例如複雜的 JSON 解析或大容量浮點運算。這會卡住主程式太久。

不可使用 Serial.print()

I/O 操作太慢，會干擾中斷時序，甚至導致 Watchdog 重新啟動。

⚡ 加速關鍵：IRAM_ATTR

使用 IRAM_ATTR 關鍵字，可以確保 ISR 程式碼存放在 RAM 中，而不是從較慢的 Flash 讀取，確保中斷反應即時。

void IRAM_ATTR onTimer() {
  // 內容越精簡越好
}

🛡️ 3. 變數共享：volatile 關鍵字 (20 min)

核心概念：當主程式與 ISR 都會用到同一個變數（如：計數器）時，必須加上 volatile。

這會防止編譯器為了優化速度，而把變數存在暫存器中「懶得讀取」，導致主程式讀到舊的數值。

講師手冊 (Instructor Guide)

教學重點 (Teaching Points)

  • 硬體優先權：解釋為什麼定時器中斷可以「強行暫停」loop 的執行。
  • ISR 的規範：強烈要求不要在 ISR 中放入 Serial.print 或 delay（會導致系統鎖死）。
  • volatile 的含義：為什麼跨線程/中斷共用的變數需要這個關鍵字（防止編譯器優化導致的舊值問題）。

學生常見問題 (Common Issues)

  • 程式一直重啟 (Watchdog Timer Error)：通常是在 ISR 裡面做了太多事情，佔用 CPU 太久。
  • 忘記附加中斷 (timerAttachInterrupt)：導致定時器設定了但永遠不會觸發。
  • 變數同步問題：loop 中讀到的計數器數值與 ISR 中的實際值不符。

評分標準 (Grading)

  • 優 (90-100): 能編寫出精簡且安全的 ISR，正確使用 volatile 並實現精確的時間控制。
  • 可 (70-89): 定時器能運作，但 ISR 過於肥大或忽略了 volatile 導致數據不一致。
  • 待加強 (Below 70): 無法成功配置硬體定時器，或程式因中斷邏輯錯誤而頻繁崩潰。
