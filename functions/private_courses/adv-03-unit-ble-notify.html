🔄 1. 觀念翻轉：從「我要數據」到「有數據通知我」 (25 min)

📥 Read (拉 - Pull)

客戶端（網頁）定時發送請求問：「現在座標是多少？」。

代價：即使數據沒變，也要消耗無線電頻寬，產生大量無效通訊。

📤 Notify (推 - Push)

客戶端先「訂閱 (Subscribe)」某個特徵值，當數值改變時，ESP32 主動推送封包。

優勢：低延遲、省電、僅在必要時占用頻寬。

🤔 思考任務

思考為什麼「按鈕按下」或「感測器警報」絕對應該使用 Notify 而非 Read？

⚙️ 2. CCCD 與 MTU 的核心限制 (20 min)

CCCD (Client Characteristic Configuration Descriptor)

Notify 不是預設開啟的。網頁端必須寫入一個特殊的描述符（CCCD），告訴 ESP32：「我已經準備好接收通知了」。沒有這個「握手」，Notify 就不會運作。

MTU (Maximum Transmission Unit)

單個 Notify 封包的大小限制（預設通常為 20-23 Bytes），這決定了我們 JSON 數據打包的上限。如果要傳大數據，就必須「切片」。

🎯 學習目標

  • 繪製 BLE 通訊中的客戶端與服務端互動流程圖。
  • 理解 Client 訂閱 CCCD 的完整過程。

⚡ 3. 事件驅動的程式流程 (15 min)

非阻塞監聽：觀察 Web Bluetooth API 中的 characteristicvaluechanged 事件。這與傳統的 setInterval 獲取數據有何本質上的不同？

🚀 預習任務：代碼觀察 

characteristic.startNotifications().then(_ => {
  characteristic.addEventListener('characteristicvaluechanged', handleNotifications);
});

當 ESP32 端呼叫 notify() 時，JS 端的回呼函式會「立即」被喚醒。

作業指引 (Assignment Guide)

作業 1-1：心跳偵測系統

  • 角色扮演： IoT 系統工程師。
  • 執行方式： 在 ESP32 端實作一個計時器，每隔 100ms 呼叫一次 `notify()` 發送模擬的感測器數據（如心跳值或座標），網頁端則使用 `characteristicvaluechanged` 事件即時更新 UI。
  • 評分重點： 能否成功開啟 Notify 訂閱，並在網頁端不重整的情況下看到數據跳動。

作業 1-2：變動觸發優化

  • 角色扮演： 韌體優化工程師。
  • 執行方式： 修改推送策略：只有當數值變化超過特定閾值 (Threshold，例如座標位移 > 5 像素) 時才發送 Notify。
  • 評分重點： 提交一份分析報告，說明此策略對「降低無線電佔空比」與「節省電力」的貢獻。

作業 1-3：大數據分段重組

  • 角色扮演： 通訊協議開發者。
  • 執行方式： 嘗試發送一段超過 20 Bytes 的 JSON 字串，實作 ESP32 端的數據切片 (Chunking) 與網頁端的緩衝區重組邏輯。
  • 評分重點： 字串拼接是否正確，且能成功被 `JSON.parse()` 解析。
