🗺️ 為什麼需要分區表？ (20 min)

ESP32 的 Flash（通常為 4MB）必須同時存放多種數據類型。

空間規劃

想像 Flash 是一個大硬碟，分區表就是分開 `C:` 碟、`D:` 碟的說明書。它告訴系統：哪個功能的數據從哪個「偏移地址 (Offset)」開始，長度是多少。

  • App： 執行的二進制程式碼。
  • Bootloader： 開機引導程式。
  • NVS： 存放 Wi-Fi 密碼、PID 參數等小數據。

常見分區類型

nvs 系統小數據
otadata OTA 狀態標記
app0 / app1 雙程式備份
spiffs / littlefs 文件系統(網頁/圖)

🛠️ 預設 vs. 自定義分區 (25 min)

為什麼有時候程式寫太大會出現 `Region iram0_0_seg' overflowed`？

Default 限制： 預設分區通常只留 1MB 給程式。加入大型影像處理庫後，空間會立刻不足。

CSV 檔案定義

學習 `partitions.csv` 的欄位格式：

# Name, Type, SubType, Offset, Size, Flags
nvs, data, nvs, , 0x4000,
otadata, data, ota, , 0x2000,
factory, app, factory, , 2M,
storage, data, spiffs, , 1M,

作業指引 (Assignment Guide)

1. GitHub Classroom 設定

  • 點擊儀表板「領取作業」按鈕，進入 GitHub Classroom 並獲取 `basic-02-partitions`。

2. 翻轉課堂學習策略

  • 課前預習：觀看「ESP32 Flash 分區表解析」影片，認識 NVS 與 SPIFFS。
  • 課中實作：修改分區大小並重新燒錄，觀察 Flash 剩餘空間的變化。
  • 課後思考：為什麼我們不把所有空間都給 APP 區？檔案系統 (SPIFFS) 的存在解決了什麼問題？

3. 單元實作任務

  • 作業 2-1 (40%)：自定義一個分區表，新增一個名為 `my_data` 的自選分區。
  • 作業 2-2 (30%)：實作 NVS (Non-volatile Storage) 讀寫，紀錄設備的開機次數。
  • 作業 2-3 (30%)：使用 SPIFFS 儲存一個簡單的檔案，並在程式啟動時讀取內容印出。

講師手冊 (Instructor Guide)

1. 教學重點 (Teaching Points)

  • 解釋 Flash 映射與虛擬記憶體的概念。
  • 介紹 Wear Leveling (磨損均衡) 機制，為什麼 Flash 不能無限次寫入。
  • 對比 SPIFFS, LittleFS 與 FATFS 的優缺點（如：檔案數量、系統穩定性）。

2. 常見問題 (Common Issues)

  • 分區表衝突 (Overlap)：提醒學生地址計算必須精確， offsets 不能重疊。
  • 找不到檔案系統：燒錄程式後，往往忘記執行 "Upload Filesystem Image" 命令。
  • NVS 損毀：解釋如果 `nvs_flash_init()` 失敗，需要先執行 `nvs_flash_erase()` 再初始化。

3. 評分標準 (Grading Criteria)

  • 基礎分：能正確自定義分區並成功儲存與讀取資料。
  • 進階分：能解釋 `0x10000` 為何是 APP 分區的慣用起始地址（對齊要求）。

在 GitHub Classroom 的實戰部署建議

A. 範本倉庫 (Template Repo) 結構

.
├── src/             # 包含 NVS 與 SPIFFS 檔案讀寫邏輯的 main.cpp
├── data/            # 包含要燒錄進檔案系統的初始檔案 (SPIFFS Upload)
├── partitions.csv   # 自定義 Flash 分區表檔案
├── README.md        # 實驗紀錄：分區空間配置計算表、燒錄成功日誌與檔案讀取截圖
└── .github/         # 自動評核配置

B. 自動評估 (Autograding) 的設計

  • 分區語法校驗： 透過測試腳本解析 partitions.csv，驗證 Offset 地址是否重疊 (Overlap) 且大小為 4KB 對齊。
  • 程式庫關聯檢查： 掃描程式中對 SPIFFS.begin() 或 Preferences.h 的使用，確認學生是否有正確實作持久化儲存。

GitHub Classroom 詳細操作指南

第一步：準備範本倉庫 (Template Repo)

建立包含基礎檔案讀寫框架但「分區表損毀」或「不完整」的 Repo，於 GitHub 設定中勾選 "Template repository"。

第二步：建立 Classroom 作業

建立 Individual Assignment，連結範本並啟用 "Feedback pull requests" 以便對其 Flash 空間規劃與磨損均衡 (Wear Leveling) 概念進行點評。

第三步：學生端操作 (Flipped Workflow)

學生領取作業，獲得個人專屬 Repo，修正分區衝突、燒錄檔案系統鏡像檔並成功讀取資料後 Push。

第四步：教師回饋 (Review)

進入 Feedback PR，針對學生的「分區優先級分配」與「檔案系統可靠性」予以回饋。
