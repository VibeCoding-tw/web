🗺️ 為什麼需要分區表？ (20 min)

ESP32 的 Flash（通常為 4MB）必須同時存放多種數據類型。

空間規劃

想像 Flash 是一個大硬碟，分區表就是分開 `C:` 碟、`D:` 碟的說明書。它告訴系統：哪個功能的數據從哪個「偏移地址 (Offset)」開始，長度是多少。

  • App： 執行的二進制程式碼。
  • Bootloader： 開機引導程式。
  • NVS： 存放 Wi-Fi 密碼、PID 參數等小數據。

常見分區類型

nvs 系統小數據
otadata OTA 狀態標記
app0 / app1 雙程式備份
spiffs / littlefs 文件系統(網頁/圖)

🛠️ 預設 vs. 自定義分區 (25 min)

為什麼有時候程式寫太大會出現 `Region iram0_0_seg' overflowed`？

Default 限制： 預設分區通常只留 1MB 給程式。加入大型影像處理庫後，空間會立刻不足。

CSV 檔案定義

學習 `partitions.csv` 的欄位格式：

# Name, Type, SubType, Offset, Size, Flags
nvs, data, nvs, , 0x4000,
otadata, data, ota, , 0x2000,
factory, app, factory, , 2M,
storage, data, spiffs, , 1M,

課程教材 - 內部使用

作業指引 (Assignment Guide)

作業 1-1：「擴容手術」自定義 partitions.csv

  • 角色扮演： 韌體架構師 (Firmware Architect)。
  • 執行方式： 重新規劃 Flash 空間。將預設的 factory 分區擴大至 2MB，並配合縮減 spiffs 分區。建立一個名為 my_data 的自選分區（size: 0x1000）。
  • 評分重點 : partitions.csv 語法正確且 Offset 地址無重疊。

作業 1-2：「數據黑盒子」NVS 參數持久化

  • 角色扮演： 嵌入式開發工程師 (Embedded Developer)。
  • 執行方式： 使用 Preferences 庫實作開機計數器與 PID 參數儲存。驗證即使在完全斷電重啟後，數據仍能正確讀回。
  • 評分重點 : 掌握 Key-Value 存取邏輯與 preferences.begin() 的正確使用。

作業 1-3：「網頁伺服器載入」LittleFS 檔案部署

  • 角色扮演： 全端 IoT 開發者 (Fullstack IoT Developer)。
  • 執行方式： 將遙控器前端檔案（HTML/CSS）存入 LittleFS 分區。修改 ESP32 程式實作「動靜態分離」，從 Flash 讀取檔案內容並透過 WebServer 回傳。
  • 評分重點 : 成功掛載 LittleFS 並透過瀏覽器存取到存儲於 Flash 的靜態網頁。

GitHub Classroom 詳細操作指南

第一步：準備範本倉庫 (Template Repo)

  1. 在 GitHub 建立新 Repo（名為 basic-02-partition-template）。
  2. 上傳包含基礎檔案讀寫框架但「分區表 configuration 損毀或不對齊」的 Repo 框架。
  3. 在 Repo 設定中勾選 "Template repository"。

第二步：建立 Classroom 作業

  1. 進入 GitHub Classroom 建立「Individual Assignment」。
  2. 連結上述的範本倉庫，並勾選 "Enable feedback pull requests"。

第三步：學生端操作 (Flipped Workflow)

  1. 學生領取作業，獲得個人專屬 Repo。
  2. 修正分區衝突，實作 SPIFFS/Preferences 存取邏輯並成功持久化資料。
  3. 本地開發後 Push 代碼 提交進度。

第四步：教師回饋 (Review)

  1. 進入 Feedback PR 的 Files Changed。
  2. 針對學生的「分區地址計算精確度」與「NVS 異常狀態處理」予以具體點評。

C. 自動評分 (Autograding) 的設計

  • 語法校驗：： 解析 partitions.csv，驗證 Offset 地址是否重疊且 Size 是否符合 4KB 對齊要求。
  • 邏輯健壯性：： 搜尋 main.cpp 中是否包含 Preferences 的正確初始化邏輯。
