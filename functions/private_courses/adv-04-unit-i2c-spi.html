通訊協議解密
I2C、SPI 與硬體實務

掌握硬體對話的本質：從定址機制、時鐘極限到數據封裝的完整旅程。

📢 I2C：省線的「廣播系統」

  • ● 僅需兩根線：SDA（數據）與 SCL（時鐘）。
  • ● 定址機制：主機透過廣播「地址」來決定與哪個設備通訊。
  • ● Open-Drain 架構：理解為什麼 I2C 總線需要上拉電阻。

⚡ SPI：高速的「專線系統」

  • ● 通常四根線：SCK、MOSI、MISO、CS（片選）。
  • ● 全雙工傳輸：數據可同時雙向移動，速度極快 (MHz)。
  • ● 點對點機制：主機利用專屬的 CS 線啟動特定設備。

📈 時序圖判讀與訊號擷取

開始位元 (Start Condition)

I2C 的獨家通訊協議：當 SCL 為高電平時，SDA 從高變低，標誌著對話的開始。

時鐘極性與相位 (CPOL/CPHA)

SPI 的四種模式。決定數據是在時鐘的「上升沿」還是「下降沿」被固定並讀取。

💡 Datasheet 尋寶：嘗試找出 MPU6050 的預設 I2C 地址，以及它在通訊時的最高時鐘頻率限制。

作業指引 (Assignment Guide)

作業 1-1：I2C 設備大掃描

  • 角色扮演： 韌體除錯工程師。
  • 執行方式： 實作一個迴圈，從 1 掃描到 127，嘗試啟動每個 I2C 地址的通訊。記錄回傳 `ACK` 的地址，並與 Datasheet 對比確認設備身份。
  • 評分重點： 能否正確找出 OLED (通常 0x3C) 與 MPU6050 (通常 0x68) 的地址。

作業 1-2：SPI 刷新率挑戰

  • 角色扮演： 顯示驅動開發者。
  • 執行方式： 逐步提升 SPI 時鐘頻率 (1MHz -> 10MHz -> 40MHz)，觀察 OLED 畫面的刷新速度，並紀錄在高頻通訊下是否出現影像雜訊或死機現象。
  • 評分重點： 實驗報告是否能指出「佈線長度」與「訊號完整性」的關聯。

作業 1-3：協議翻譯官：Raw to JSON

  • 角色扮演： 嵌入式系統架構師。
  • 執行方式： 從 I2C 感測器讀取 6 Bytes 的原始二進制數據，使用物件結構 (Struct) 解析後，再利用 ArduinoJson 轉換成標準 JSON 格式發送。
  • 評分重點： 數據轉換的過程是否準確，且 JSON 欄位命名是否具備語義化。
