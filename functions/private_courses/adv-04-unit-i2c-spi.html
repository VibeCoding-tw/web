進階 4 - 單元 1

通訊協議解密
I2C、SPI 與硬體實務

掌握硬體對話的本質：從定址機制、時鐘極限到數據封裝的完整旅程。

📢 I2C：省線的「廣播系統」

  • ● 僅需兩根線：SDA（數據）與 SCL（時鐘）。
  • ● 定址機制：主機透過廣播「地址」來決定與哪個設備通訊。
  • ● Open-Drain 架構：理解為什麼 I2C 總線需要上拉電阻。

⚡ SPI：高速的「專線系統」

  • ● 通常四根線：SCK、MOSI、MISO、CS（片選）。
  • ● 全雙工傳輸：數據可同時雙向移動，速度極快 (MHz)。
  • ● 點對點機制：主機利用專屬的 CS 線啟動特定設備。

📈 時序圖判讀與訊號擷取

開始位元 (Start Condition)

I2C 的獨家通訊協議：當 SCL 為高電平時，SDA 從高變低，標誌著對話的開始。

時鐘極性與相位 (CPOL/CPHA)

SPI 的四種模式。決定數據是在時鐘的「上升沿」還是「下降沿」被固定並讀取。

💡 Datasheet 尋寶：嘗試找出 MPU6050 的預設 I2C 地址，以及它在通訊時的最高時鐘頻率限制。

硬體實戰任務 (Labs)

掌握硬體通訊的底層邏輯與數據轉換。

🔍 60 min

【作業 1-1】I2C 設備大掃描

任務目標：在 ESP32 上掃描連接的所有 I2C 設備，找回硬體定址。

  • 實作 I2C 通訊協議初始化
  • 找回設備地址 (e.g., 0x3C, 0x68)
  • 同時驅動 OLED 與加速度計
⚡ 60 min

【作業 1-2】SPI 刷新率挑戰

任務目標：測試 SPI 高頻極限，理解訊號完整性。

  • 調整 SPI 頻率 (1MHz - 40MHz)
  • 記錄影像雜訊出現的臨界點
  • 分析高頻通訊下的波形失真
📦 45 min

【作業 1-3】協議翻譯官

任務目標：將硬體 Raw Data 轉換為現代 Web 協議格式。

  • 使用 ArduinoJson 封裝數據
  • 建立 I2C 數據與 JSON 結構的對映
  • 為網頁端提供異步通訊接口
