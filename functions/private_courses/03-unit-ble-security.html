Module 1: Intuitive Lab

安全沙箱：為什麼我的連線按鈕沒反應？

1 安全上下文 (Secure Context)

Web Bluetooth 是一個強大的 API，它允許網頁直接控制實體設備。為了防止惡意網站偷偷連線你的設備，瀏覽器強制要求網站必須在 HTTPS 環境下才能執行。

🔓

HTTP (不安全)

navigator.bluetooth 為 undefined

🔒

HTTPS (安全)

navigator.bluetooth 可正常呼叫

console
// 在 http://localhost 執行時
Uncaught TypeError: Cannot read properties of undefined (reading 'requestDevice')
at connectBtn.onclick (main.js:12)
// 在 https://mysite.com 執行時
> scanning for devices...
🛡️
Rule #1
必須是 HTTPS
Module 2: Core Principles

Web BLE 三大安全鐵律

🔐

1. 環境限制

除了 localhost 用於開發測試外，所有上線的 Web App 必須部署在具備 SSL 憑證 (HTTPS) 的伺服器上。

  • ✔ localhost
  • ✔ 127.0.0.1
  • ✔ https://...
  • ✘ http://192.168.x.x
👆

2. 使用者手動觸發

瀏覽器禁止網頁在背景「自動」掃描。所有的連線請求必須由使用者明確的點擊 (User Gesture) 觸發。

btn.addEventListener('click', function() {
  // 必須在這裡呼叫
  navigator.bluetooth.requestDevice(...)
});
🔎

3. 設備過濾與隱私

網頁不能掃描「所有」設備。必須明確宣告要存取的 filters 或 optionalServices，否則瀏覽器會拒絕執行。

隱私保護：
這防止了廣告商透過掃描你身邊的藍牙設備 (如智慧手環) 來追蹤你的位置。

1. 為什麼適合在 GitHub Classroom 進行？

作業 1-1：try-catch 機制（邏輯優化）

  • GitHub 優勢：這是標準的代碼品質測試。你可以透過自動評分檢查學生的 index.js 是否包含 try-catch 區塊，以及是否正確判斷了 NotFoundError。
  • 翻轉教學：學生在課前練習「失敗處理」。課堂上，老師可以關掉藍牙或故意點取消，測試誰的程式碼會「當掉」，誰的會「優雅提示」。

作業 1-2：精準過濾（實務應用）

  • GitHub 優勢：適合練習 環境變數或組態管理。你可以要求學生在代碼中定義一個 CONFIG 物件來存放過濾條件。
  • 翻轉教學：這是課堂實作的高潮。全班 20 台車同時開啟，若學生沒做好過濾，會連到別人的車。這是一個極佳的「現場除錯」教學點，學生可以直接在 GitHub 更新代碼並重新部署（如使用 GitHub Pages）來即時修正。

作業 1-3：跨平台調查（文件化與協作）

  • GitHub 優勢：適合練習使用 Markdown (README.md) 或 GitHub Wiki。這份作業不需要寫太多程式碼，而是要學生紀錄觀察結果。
  • 翻轉教學：學生將不同裝置的配對截圖傳到 screenshots/ 資料夾。課堂上可以利用 GitHub 的 Pull Request 比較功能，快速投放在大螢幕上對比 Android 與 iOS 的介面差異。

2. 在 GitHub Classroom 上的設定建議

針對這系列 Web BLE 任務，建議將它們合併為一個名為 BLE-Connection-Lab 的 Assignment。

A. 範本倉庫建議結構
.
├── index.html       # 包含連線按鈕與訊息顯示區域
├── app.js           # 學生主要實作區 (1-1, 1-2)
├── screenshots/     # 存放作業 1-3 的跨平台截圖
└── README.md        # 紀錄 1-3 的調查結果與排除心得
B. 自動評分 (Autograding) 配置

你可以針對 1-1 與 1-2 設定簡單的字串檢查：

  • 測試 1 (1-1)：檢查是否使用了 catch 與 NotFoundError。
    指令：grep "catch" app.js && grep "NotFoundError" app.js
  • 測試 2 (1-2)：檢查是否使用了 filters 或 optionalServices。
    指令：grep "filters" app.js || grep "namePrefix" app.js

3. 給老師的翻轉課堂執行建議

階段 任務內容 老師如何利用 GitHub
課前 (自主學習) 閱讀 Web BLE 文件，完成 1-1 的錯誤處理邏輯。 查看學生的 Commit History，確認誰已經開始動手寫。
課中 (實作對抗) 進行 1-2 的「多車連線挑戰」。學生必須修改過濾條件以連上自己的 ESP32。 Live View：透過 GitHub Pages，學生修改完 Push 後，手機重新整理即可測試新邏輯。
課後 (總結) 整理 1-3 的跨平台差異，提交最終報告。 Code Review：老師在學生提交的 PR 下方給予回饋，特別是針對錯誤處理的完整度。

這組作業非常有「現場感」，特別是 1-2 的過濾功能，在教室環境下是非常強大的教學工具。
