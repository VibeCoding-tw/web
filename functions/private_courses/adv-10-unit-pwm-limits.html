限制與飽和：
PWM 的物理極限

當數學計算遇上物理極限，如何保持幾何軌跡的完美？

🛑 1. 什麼是 PWM 飽和？ (15 min)

  • ● 物理極限：ESP32 的 PWM 通常設定為 8-bit（0-255）。當你的 PID 或幾何公式算出來需要 300 的電力時，馬達只能給你 255。
  • ● 失去比例 (Loss of Ratio)：在差速驅動中，轉向靠的是左右輪的「比例」。
    如果左輪需要 300（飽和成 255），右輪需要 150。
    原本 2:1 的比例變成了 255:150 ≈ 1.7:1，車子就會轉不過彎或偏離預定弧度。

⚠️ 警示

不當的飽和處理（如直接 Clipping）會直接破壞幾何模型的準確性，導致導航誤差。

Visualization of Ratio Loss

👻 2. 死區 (Dead Zone) 現象 (15 min)

啟動摩擦力：並非 PWM 給 1 馬達就會轉。通常在 PWM 低於某個數值（如 40）時，馬達只會發出吱吱聲而不動。

  • ● 有效區間：理解馬達的實際作業區間僅在 DeadZone < PWM < 255 之間。
  • ● 非線性響應：在低速區間，馬達輸出的扭力往往不穩定，需要透過軟體補償（如 Minimal PWM）來解決。
0 ... 40 ... 255 Dead Zone Area (No Motion)

⚖️ 3. 等比例縮放演算法 (Scaling Algorithm) (30 min)

思考如何解決飽和問題？

方法 A (Clipping) - ❌ 錯誤

直接把超過 255 的變成 255。這樣會破壞左右輪的速度比例，導致車輛轉向半徑改變。

方法 B (Scaling) - ✅ 正確

找出左右輪中最大的一個，如果它超過 255，則將兩輪同時除以一個係數，使最大的一輪剛好等於 255。

🔢 練習計算

若左輪算出來是 400，右輪是 200。請計算使用「方法 B」後，兩輪實際的 PWM 應該是多少？

查看答案

Max = 400. Factor = 255 / 400 = 0.6375.
Left = 400 * 0.6375 = 255
Right = 200 * 0.6375 = 127.5
比例維持 2:1。

作業指引 (Assignment Guide)

作業 3-1：「飽和失控」觀測實驗

  • 角色扮演： 系統分析師。
  • 執行方式： 故意在代碼中輸入不合理的參數（如：超高車速 \(V\) 搭配小轉彎半徑 \(R\)）。先使用簡單的 `constrain(pwm, 0, 255)` 處理輸出。
  • 評分重點： 在 README 中記錄並分析為什麼「直接裁剪 (Clipping)」會導致車子跑出奇怪的軌跡（例如：原本要轉彎卻變成直線）。理解「比例」被破壞的後果。

作業 3-2：「比例保衛戰」：實作等比例縮放 (Scaling)

  • 角色扮演： 運動控制演算法工程師。
  • 執行方式： 實作 `normalizePWM(left, right)` 函式。若任一輪需求大於 255，則將兩者同時除以一個縮放係數，使最大者剛好等於 255。
  • 評分重點： 重新執行 3-1 的測試。觀測即使在「飽和」狀態下，車子是否仍能維持正確的轉彎半徑（僅是速度變慢）。

作業 3-3：克服死區 (Deadzone) 與線性化補償

  • 角色扮演： 硬體調校專家。
  • 執行方式： 慢慢增加 PWM 數值，記錄馬達剛好能轉動的最小值 \(PWM_{min}\)。使用 `map()` 函式將邏輯上的 0-255 映射到實際的 \(PWM_{min}\) 到 255。
  • 評分重點： 調校後的車子在極低速下是否能流暢行駛，而非發出嗡嗡聲卻卡住。
