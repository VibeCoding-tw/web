基礎 10 - 單元 2

網頁端模式控制
介面設計與安全實務

設計一個直覺、零失誤且與硬體同步的戰術控制面板。

🖱️ 1. 模式切換的 UI 邏輯 (30 min)

控制組件選擇：介面的直覺性來自於選擇正確的控制項。

Segmented Control

適合 2-3 個互斥模式（如：手控 / 自動）。所有選項攤開，當前選中項高亮。

Toggle Switch

適合單一功能的開啟/關閉（如：避障功能 ON/OFF）。直觀模擬物理開關。

視覺層級 (Visual Hierarchy)

  • 顏色語義：紅色代表「緊急/停止」、藍色代表「手控模式」、綠色代表「全自動」。
  • 狀態顯眼化：當前模式應有明顯的背景色或陰影（.active 類別的視覺呈現）。

📱 2. 響應式佈局與防誤觸 (30 min)

CSS Flexbox/Grid

確保按鈕在不同尺寸的手機上都能輕鬆被大拇指點擊。使用 min-height: 44px 或更大的觸控熱區。

防誤觸設計

模式切換按鈕不應緊貼著「前進/後退」搖桿，避免操作時誤觸導致模式突變，造成安全隠憂。

💡 實務技巧

在自動駕駛模式下，應將手動控制項（如搖桿）變為「禁用狀態」或降低透明度，讓使用者明確知道目前操作權已交給系統。

🔄 3. 預習：資料雙向綁定 (20 min)

核心概念：硬體狀態與網頁介面必須始終如一。

狀態自動校正邏輯

當車子因為感測器故障「自動切回待機」時，網頁上的按鈕也要透過 WebSocket/BLE Notify 跟著變動，而非只是單向發送指令。

關鍵概念：
單選按鈕 (Radio Buttons) 與切換開關 (Toggle Switch) 的前端實作。
學習核心：
使用 JavaScript 的 fetch() 或 WebSocket 發送「模式切換」指令給 ESP32。
// JavaScript 狀態同步範例
function updateUI(hardwareMode) {
  // 移除所有模式的 active 類別
  document.querySelectorAll('.mode-btn').forEach(btn => btn.classList.remove('active'));

  // 根據硬體回傳值設定對應按鈕
  const targetBtn = document.getElementById(`btn-${hardwareMode}`);
  if (targetBtn) {
    targetBtn.classList.add('active');
    document.body.style.backgroundColor = (hardwareMode === 'AUTO') ? '#f0fdf4' : '#f8fafc';
  }
}

核心任務：設計「零失誤」戰術面板

透過 UI/UX 上的巧思，讓使用者在操作複雜機器時也能得心應手。

🎨 60 min

【作業 2-1】

「戰術面板」：CSS 狀態切換

  • 三模面板：設計 Manual、Auto-Pilot、Eco-Mode 選項。
  • 動態樣式：利用 JS 切換 CSS 類別（例如 .active），讓按鈕發光或變色。
  • UI 鎖定：當切換至 Auto-Pilot 時，虛擬搖桿應變為「半透明」且「無法觸控」。
🔄 60 min

【作業 2-2】

雙向狀態同步 (WebSocket/BLE)

  • 心跳同步：每秒從 ESP32 發送一次當前 FSM 狀態給網頁。
  • 自動校正：如果收到的狀態與網頁顯示不符，網頁應自動修正外觀。
  • 安全隱憂：解決「網頁顯示手動，但機器正在自動」的不一致危險。
🚨 45 min

【作業 2-3】

緊急停機與視覺警告

  • 滿版遮罩：點擊緊急鈕時，彈出一個紅色半透明遮罩，鎖死所有操作。
  • 觸覺反饋：呼叫 navigator.vibrate() 讓手機震動提醒。
  • 多感官回饋：利用視覺、觸覺確保危險發生時的即時警示。

講師手冊 (Instructor Guide)

教學重點 (Teaching Points)

  • 控制項的語意化：解釋何時該用 Segmented Control，何時該用 Toggle Switch。
  • 防誤觸設計：探討 UI 佈局如何影響操作安全（如：不要把「自爆」按鈕放在「加速」按鈕旁邊）。
  • 多重感官回饋：強調在戶外強光下，單純的顏色變化可能不夠，需要震動或彈出遮罩。

學生常見問題 (Common Issues)

  • 觸控熱區太小：在手機上很難點中特定的按鈕（建議 min-height: 44px）。
  • 狀態回彈：當硬體還沒切換成功前，UI 就先跳過去了（討論回歸 SSoT 的重要性）。
  • 忘記禁用受限操作：在自動駕駛模式下仍然允許編輯路徑參數（可能引發崩潰）。

評分標準 (Grading)

  • 優 (90-100): UI 極致精美流暢，具備清晰的視覺層級，且安全機制（如 E-Stop）實作完善。
  • 可 (70-89): 基本控制功能正常，但 UI 美感不足，或在手機上的操作便利性待提升。
  • 待加強 (Below 70): 介面凌亂，無法建立起使用者與硬體之間的直覺連結。

  • 邊界反應速度： 驗證按下 E-Stop 按鈕後，是否立即向硬體發送高優先權停機指令。

GitHub Classroom 詳細操作指南

第一步：準備範本倉庫 (Template Repo)

建立包含基礎戰術面板樣式的 Repo，於 GitHub 設定中勾選 "Template repository"。

第二步：建立 Classroom 作業

建立 Individual Assignment，連結範本並啟用 "Feedback pull requests" 以便對其 UI 佈局與事件處理進行點評。

第三步：學生端操作 (Flipped Workflow)

學生領取作業，獲得個人專屬 Repo，實作 UI 同步邏輯與震動回饋並 Push 觀察其在不同裝置上的呈現。

第四步：教師回饋 (Review)

進入 Feedback PR，針對學生的「多重感官回饋」設計巧思給予專業建議。
