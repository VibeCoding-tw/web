🔄 OTA 的核心：雙分區機制 (20 min)

如何確保更新過程中斷電，機器也不會壞掉？

1

預防「磚頭化」 (Bricking)

絕不能直接覆蓋正在執行的程式！OTA 採用「先下載到備份區，驗證成功後再切換」的策略。

2

分區切換 (A/B Partition)

ESP32 將 Flash 分為 app0 與 app1。當 app0 正在執行時，OTA 會將新韌體寫入 app1。反之亦然。

3

驗證與啟動標記

寫入完成後會檢查 Checksum。成功後修改 `otadata` 分區，指引下次開機從新分區啟動。

🌐 Web OTA 的傳輸流程 (25 min)

1. HTTP POST

Web OTA 本質上是將 .bin 檔案透過瀏覽器的 `FormData` 上傳到 ESP32 的特定路由。

2. 串流寫入

ESP32 收到數據時，直接調用 `Update.write()` 將二進制流寫入 Flash 的特定地址。

3. 進度回報

透過 JavaScript 的 `XMLHttpRequest` 監聽 `progress` 事件，實現視覺化進度條。

browser-upload.js

前端進度監聽範例

xhr.upload.addEventListener('progress', function(e) {
  if (e.lengthComputable) {
    const percent = Math.round((e.loaded / e.total) * 100);
    progressBar.style.width = percent + '%';
    statusText.innerText = `上傳中: ${percent}%`;
  }
});

🔐 安全與驗證 (15 min)

如果隨便一個連上 Wi-Fi 的人都能上傳韌體，會發生什麼事？安全機制是生產等級產品的必修課。

  • ✔ 身份驗證： 在更新頁面前增加密碼彈窗或 Token 驗證。
  • ✔ mDNS (網路發現)： 讓設備在網路上以 `esp32.local` 身份出現。
  • ✔ ElegantOTA： 解析此庫如何封裝 驗證 -> 寫入 -> 切換 的過程。

講師手冊 (Instructor Guide)

1. 教學重點 (Teaching Points)

  • 解釋 ESP32 的啟動引導程式 (Bootloader) 如何判斷應該執行哪一個 Partition。
  • 討論 HTTP 傳輸過程中的完整性校驗 (MD5/SHA256)。
  • 強調「優雅失敗」的重要性：確保設備即使在升級中斷後也不會變磚。

2. 常見問題 (Common Issues)

  • 空間不足 (Magic signature mismatch)：檢查 Partition Table 分配是否給予 OTA 槽位足夠的大小。
  • 連線中斷：如果是校園網，請確認 AP 偵測是否禁止設備間通訊 (AP Isolation)。
  • 版本衝突：確保燒錄進去的韌體有確實修改版本號字串，以供辨識。

3. 評分標準 (Grading Criteria)

  • 基礎分：能成功透過 Web 介面完成一次韌體更新。
  • 進階分：能完整描述 `esp_ota_mark_app_valid_cancel_rollback()` 的調用時機。

GitHub Classroom 詳細操作指南

第一步：準備範本倉庫 (Template Repo)

建立包含基礎 WiFi 連線但「不具備 OTA 功能」的 Repo，於 GitHub 設定中勾選 "Template repository"。

第二步：建立 Classroom 作業

建立 Individual Assignment，連結範本並啟用 "Feedback pull requests" 以點評學生的網路安全配置與回滾 (Rollback) 策略。

第三步：學生端操作 (Flipped Workflow)

學生領取作業，獲得個人專屬 Repo，實作空中升級介面並透過無線方式成功更新韌體後 Push。

第四步：教師回饋 (Review)

進入 Feedback PR，針對學生的「升級狀態回饋 (UI Feedback)」與「異常中斷處理」予以回饋。
