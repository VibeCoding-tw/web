進階 13 - 單元 2

系統性能分析與瓶頸優化

打破速度的天花板：從硬體運算、通訊協議到前端渲染的全鏈路優化。

為什麼要優化性能？

系統的速度取決於「最慢」的那一環。即使你有最強的馬達，如果 ESP32 還在慢慢 loop，或者藍牙傳輸卡住，車子的反應就會像「lag」一樣遲鈍。

本單元目標

  • ✓辨識三大瓶頸源：通訊、運算、渲染
  • ✓學會使用 micros() 與 DevTools 測量延遲
  • ✓實作指令瘦身與 Canvas 渲染優化

1 辨識性能瓶頸 (Bottlenecks)

要解決「慢」的問題，首先要找出「哪裡慢」。在 IoT 系統中，瓶頸通常出現在以下三個地方：

1. 通訊瓶頸

藍牙/Wi-Fi 頻寬不足或封包過大。

  • 傳送了過於冗長的 JSON
  • Web BLE 的 MTU 限制

2. 運算瓶頸

ESP32 處理不及導致阻塞。

  • loop() 裡有 delay()
  • 大量的 Serial.print()
  • 複雜的浮點數運算

3. 渲染瓶頸

網頁端繪圖過於頻繁。

  • Canvas 每秒重繪太多次
  • 垃圾回收 (GC) 頻繁觸發

2 性能觀測工具

硬體端：Code Profiling (micros)

別用猜的！在程式碼中埋設「時間點」，計算一段程式碼跑了多少微秒 ($\mu s$)。

void loop() {
    unsigned long start = micros(); // 記錄開始時間
    
    // ... 執行你的任務 ...
    doComplexMath();
    
    unsigned long duration = micros() - start; // 計算耗時
    Serial.printf("Task took %lu us\n", duration);
}

前端：Chrome DevTools Performance

利用瀏覽器的開發者工具 (F12) -> Performance 面板，錄製一段操作過程，觀察 Scripting (黃色) 和 Rendering (紫色) 的時間佔比。

3 數據壓縮策略

為什麼少即是多？

減少封包長度能直接提升通訊頻率。比較以下兩種 JSON 格式：

{"linear_velocity":100, "angular_velocity":50}

約 45 bytes (冗長)

{v:100,w:50}

約 12 bytes (精簡)

甚至可以極限壓縮成二進位格式或自定義字串（如 `L100R50`），將通訊效率提升 3-4 倍。

實作任務：極速傳說

從毫秒計較到數據瘦身，壓榨系統的最後一滴性能。

作業指引 (Assignment Guide)

作業 2-1：「瓶頸偵探」：控制循環性能分析

  • 角色扮演： 系統性能分析師。
  • 執行方式： 在 ESP32 代碼中對影像處理、PID 計算、BLE 發送分別計時，並透過 Serial 印出耗時百分比。
  • 評分重點： 能否準確識別出哪一部分是系統的「木桶短板」，並提出代碼層級的優化構想（如減少不必要的浮點運算）。

作業 2-2：「輕量化大挑戰」：自定義二進位協議

  • 角色扮演： 通訊協議專家。
  • 執行方式： 放棄冗長的 JSON 格式，改用 Typed Array (Uint8Array) 封裝指令，手動進行 Byte 級別的打包。
  • 評分重點： 單次封包大小是否減少 50% 以上，以及在藍牙傳輸高壓下，原本的「斷點」現象是否消失。

作業 2-3：「性能極限測試」：壓榨硬體潛能

  • 角色扮演： 硬體優化大師。
  • 執行方式： 嘗試提升 Task 的優先級或是利用雙核心處理，將採樣時間壓低到硬體極限（如 5ms）。
  • 評分重點： 紀錄並對比優化前後的「控制延遲」數據，並附上波形穩定性的證據。
