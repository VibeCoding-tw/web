進階 13 - 單元 2

系統性能分析與瓶頸優化

打破速度的天花板：從硬體運算、通訊協議到前端渲染的全鏈路優化。

為什麼要優化性能？

系統的速度取決於「最慢」的那一環。即使你有最強的馬達，如果 ESP32 還在慢慢 loop，或者藍牙傳輸卡住，車子的反應就會像「lag」一樣遲鈍。

本單元目標

  • ✓辨識三大瓶頸源：通訊、運算、渲染
  • ✓學會使用 micros() 與 DevTools 測量延遲
  • ✓實作指令瘦身與 Canvas 渲染優化

1 辨識性能瓶頸 (Bottlenecks)

要解決「慢」的問題，首先要找出「哪裡慢」。在 IoT 系統中，瓶頸通常出現在以下三個地方：

1. 通訊瓶頸

藍牙/Wi-Fi 頻寬不足或封包過大。

  • 傳送了過於冗長的 JSON
  • Web BLE 的 MTU 限制

2. 運算瓶頸

ESP32 處理不及導致阻塞。

  • loop() 裡有 delay()
  • 大量的 Serial.print()
  • 複雜的浮點數運算

3. 渲染瓶頸

網頁端繪圖過於頻繁。

  • Canvas 每秒重繪太多次
  • 垃圾回收 (GC) 頻繁觸發

2 性能觀測工具

硬體端：Code Profiling (micros)

別用猜的！在程式碼中埋設「時間點」，計算一段程式碼跑了多少微秒 ($\mu s$)。

void loop() {
    unsigned long start = micros(); // 記錄開始時間
    
    // ... 執行你的任務 ...
    doComplexMath();
    
    unsigned long duration = micros() - start; // 計算耗時
    Serial.printf("Task took %lu us\n", duration);
}

前端：Chrome DevTools Performance

利用瀏覽器的開發者工具 (F12) -> Performance 面板，錄製一段操作過程，觀察 Scripting (黃色) 和 Rendering (紫色) 的時間佔比。

3 數據壓縮策略

為什麼少即是多？

減少封包長度能直接提升通訊頻率。比較以下兩種 JSON 格式：

{"linear_velocity":100, "angular_velocity":50}

約 45 bytes (冗長)

{v:100,w:50}

約 12 bytes (精簡)

甚至可以極限壓縮成二進位格式或自定義字串（如 `L100R50`），將通訊效率提升 3-4 倍。

📝 實作任務 (Classroom Challenges)

Task 2-1 「耗時追蹤」：硬體端分析 (60 min)

任務說明 (Mission)

測量並優化 ESP32 的主迴圈執行效率，找出拖慢系統的元兇。

  • 時間測量：在 loop() 的開頭與結尾記錄 micros()，算出每秒可以跑幾次迴圈（Hz）。
  • 瓶頸切除：嘗試移除所有的 delay() 與不必要的 Serial.print()，觀察迴圈頻率的變化（通常會提升 10 倍以上）。
  • 目標：確保硬體處理頻率遠高於控制指令頻率（例如 > 100Hz）。

Task 2-2 「指令瘦身」：通訊協議優化 (60 min)

任務說明 (Mission)

將原始 JSON 指令重構成更輕量化的格式，提升無線傳輸效率。

  • 協議重構：將 {"left":255, "right":255} 改成自定義的二進位格式或極簡字串（如 L255R255）。
  • 解析實作：在 ESP32 端撰寫對應的解析邏輯（如 sscanf 或自定義 parser）。
  • 延遲測試：對比瘦身前後，從按下按鈕到馬達轉動的反應時間。

Task 2-3 前端渲染優化 (45 min)

任務說明 (Mission)

優化網頁儀表板的繪圖效能，避免卡頓。

  • 離屏渲染 (Off-screen Rendering)：將不變的儀表背景畫在另一個 Canvas 上，只在主畫布更新變動的指針。
  • Frame Rate 限制：利用 requestAnimationFrame 確保 UI 更新頻率與螢幕刷新率（60Hz）同步，不浪費運算資源。
  • 目標：即使在低階手機上也能流暢顯示無人車狀態。
