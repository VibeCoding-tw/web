為什麼要學數據流？

開發者常常迷失在代碼細節中，卻忽略了系統的全貌。理解數據從「手指觸摸螢幕」到「馬達轉動」的完整路徑，是具備「端到端 (End-to-End)」除錯能力的關鍵。

🎯

本單元重點

  • ✓掌握感知、傳輸、執行三層架構
  • ✓理解序列化 (Serialization) 與數據映射
  • ✓體會同步與非同步的設計差異

🏗️ 1. 系統三層架構全覽 (20 min)

1. 感知/控制層 (Web UI)

捕捉觸控事件，產生原始的 (x, y) 座標數據。

Event: touchstart

2. 傳輸層 (BLE / Wi-Fi)

負責數據封裝，跨越空間飛向 ESP32。

Packet: {"cmd":"MOVE"}

3. 執行/反饋層 (ESP32)

解析封包，轉化為 PWM 訊號驅動馬達。

analogWrite(PIN, 255)

🔄 2. 數據的變身術：從 String 到 PWM (20 min)

必須將概念轉換為機器可讀格式。JSON 方便閱讀但體積大，Binary 難讀但傳輸快。

作業指引 (Assignment Guide)

作業 1-1：數據流向分析儀

  • 角色扮演： 系統整合工程師。
  • 執行方式： 在網頁 Console 埋點記錄發送時間戳記，並在 ESP32 Serial Monitor 記錄接收時間戳記，比對數據一致性。
  • 評分重點： 能否準確辨別 Byte Order (Little Endian vs Big Endian) 導致的數值解析錯誤。

作業 1-2：數據延遲與瓶頸挑戰

  • 角色扮演： 性能測試工程師。
  • 執行方式： 嘗試手動將傳輸速率提升至 100Hz，並注入巨型 JSON 字串，記錄系統何時開始出現大量封包丟失。
  • 評分重點： 提交一份延遲分析曲線，說明「封包大小」與「即時性」之間的權衡關係。

作業 1-3：系統異常模擬與防禦

  • 角色扮演： 安全性開發人員。
  • 執行方式： 設計一個能從亂碼注入中自動復原的解析器（例如：Try-Parse 邏輯）。
  • 評分重點： 當數據流中出現非預期字符時，系統是否能正常忽略非法包並繼續處理後續合法包。
