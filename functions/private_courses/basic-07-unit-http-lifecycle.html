數據的一趟旅程：
HTTP 協議與請求生命週期

拆解 HTTP 封包結構，跟隨數據包在網路森林中的跳轉，理解如何精準控制你的機器人。

📦 1. HTTP 報文結構：封包裡裝了什麼？ (20 min)

Request (請求)

  • Method

    動作類型

    GET (拿數據)、POST (傳指令)。

  • Headers

    身份證與清單

    例如 Content-Type: application/json。

  • Body

    實際載重

    例如控制馬達的 JSON 數據。

Response (回應)

  • Status Code

    執行結果

    200 (成功)、404 (找不到路徑)、500 (車子當機)。

作業指引 (Assignment Guide)

【作業 2-1】「封包捕手」：F12 開發者工具實測

  • 角色扮演： 網路診斷工程師 (Network Diagnostic Engineer)。
  • 執行方式： 利用 Chrome DevTools 的 Network 面板，分析請求的生命週期。重點觀察 TTFB (首字節時間) 與 Content Download，並截圖說明延遲是發生在伺服器運算還是網路傳輸。
  • 評分重點 : 能準確辨識請求各階段耗時，並能針對「Waiting」耗時過長提出合理的優化假設（如硬體運算過載）。

【作業 2-2】「自建路由」：ESP32 請求解析器

  • 角色扮演： 嵌入式 API 開發者 (Embedded API Developer)。
  • 執行方式： 在 ESP32 上實作自定義路由處理器。建立 /api/status (GET) 用於回傳感測器數據，以及 /api/control (POST) 用於接收控制指令。必須正確處理 HTTP 狀態碼與 JSON 回應格式。
  • 評分重點 : 路由回應邏輯清晰，JSON 格式符合 RFC 規範，且能在瀏覽器端成功獲取數據。

【作業 2-3】「連線池優化」：Keep-Alive 實驗

  • 角色扮演： 系統效能優化專家 (System Performance Specialist)。
  • 執行方式： 比較「短連線 (Connection: Close)」與「長連線 (Keep-Alive)」對頻繁指令下發的效能影響。計算在 10 秒內連續發送 50 次指令的總耗時與成功率。
  • 評分重點 : 實驗流程嚴謹，數據對比明顯，並能正確解釋 TCP 握手次數減少對物聯網通訊的優點。

GitHub Classroom 詳細操作指南

第一步：準備範本倉庫 (Template Repo)

  1. 在 GitHub 建立新 Repo（名為 basic-07-http-life-template）。
  2. 上傳包含基礎 HTTP 請求處理框架但「未實作特定路由邏輯」的專案代碼。
  3. 在 Repo 設定中勾選 "Template repository"。

第二步：建立 Classroom 作業

  1. 進入 GitHub Classroom 建立「Individual Assignment」。
  2. 連結上述的範本倉庫，並勾選 "Enable feedback pull requests"。

第三步：學生端操作 (Flipped Workflow)

  1. 學生領取作業，獲得個人專屬 Repo。
  2. 透過瀏覽器工具對比不同響應長度對連線耗時的影響，並實作 API 路由。
  3. 完成效能分析報告後 Push 代碼與 MD 報告 提交進度。

第四步：教師回饋 (Review)

  1. 進入 Feedback PR 的 Files Changed。
  2. 針對學生的「請求瓶頸識別」準確度與「響應標頭配置」予以回饋。

在 GitHub Classroom 的實戰部署建議

此單元聚焦於「網路通訊深度剖析」，引導學生從表面指令深入到底層封包的交互與效能優化。

A. 範本倉庫 (Template Repo) 結構

.
├── src/
│   └── main.cpp      # (重點) 需實作 server.on("/api/status", ...)
├── README.md        # 效能報告：Keep-Alive 對比實驗數據與 Network 面板截圖
└── .github/         # 自動評分配置

B. 參考範例代碼 (Reference Example Code)

main.cpp (Route with JSON)

server.on("/api/status", HTTP_GET, [](AsyncWebServerRequest *request){
  String json = "{\"status\":\"ok\",\"uptime\":" + String(millis()) + "}";
  request->send(200, "application/json", json);
});

C. 自動評分 (Autograding) 的設計

  • 路由可用性檢查：： 模擬發送 GET /api/status，驗證是否有正確格式的 JSON 回傳。
  • 效能分析報告審核：： 檢查 README.md 中是否包含對 TTFB 與 Keep-Alive 的量化對比數據。
