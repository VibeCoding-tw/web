數據的一趟旅程：
HTTP 協議與請求生命週期

拆解 HTTP 封包結構，跟隨數據包在網路森林中的跳轉，理解如何精準控制你的機器人。

📦 1. HTTP 報文結構：封包裡裝了什麼？ (20 min)

Request (請求)

  • Method

    動作類型

    GET (拿數據)、POST (傳指令)。

  • Headers

    身份證與清單

    例如 Content-Type: application/json。

  • Body

    實際載重

    例如控制馬達的 JSON 數據。

Response (回應)

  • Status Code

    執行結果

    200 (成功)、404 (找不到路徑)、500 (車子當機)。

講師手冊 (Instructor Guide)

教學重點 (Teaching Points)

  • 生命週期全貌：從點擊網址、DNS 解析、TCP 連線到資料呈現。
  • 延遲 (Latency) vs 吞吐量 (Throughput)：解釋為什麼對 IoT 來說，連線建立時間往往比下載速度更重要。
  • 解析 Network Waterfall 圖表：如何識別請求瓶頸。

學生常見問題 (Common Issues)

  • 不理解為什麼明明連上了 WiFi，請求還是會超時（通常是 Gateway 或 DNS 設定問題）。
  • 過度頻繁地發送請求，導致網路擁塞（解釋 Connection: Close vs Keep-Alive）。
  • 忽略了 Content-Length 的重要性，導致客戶端接收不完全。

評分標準 (Grading)

  • 優 (90-100): 能準確解讀 Network 面板各項參數，並針對硬體端進行有效率的響應優化。
  • 可 (70-89): 了解 HTTP 流程，但對於 TCP 連線細節或 DNS 影響理解較淺。
  • 待加強 (Below 70): 僅能完成基礎請求，但不理解請求在底層是如何傳遞與阻塞的。

在 GitHub Classroom 的實戰部署建議

A. 範本倉庫 (Template Repo) 結構

.
├── src/             # 包含 HTTP 請求封裝與各階段時戳紀錄的 main.cpp
├── README.md        # 數據分析報告：瀏覽器 Network Waterfall 截圖與分析
└── .github/         # 自動評分配置

B. 自動評估 (Autograding) 的設計

  • 生命週期完整性檢核： 驗證學生提交的 README 是否包含完整的請求各階段解析（DNS, TCP, Request, Waiting）。
  • 優化效果量化： 檢查提交的測試數據，驗證實作「最簡響應」後，Waiting Time (TTFB) 是否有顯著下降。

GitHub Classroom 詳細操作指南

第一步：準備範本倉庫 (Template Repo)

建立包含基礎 HTTP 請求追蹤 frameworks 的 Repo，於 GitHub 設定中勾選 "Template repository"。

第二步：建立 Classroom 作業

建立 Individual Assignment，連結範本並啟用 "Feedback pull requests" 以便對其通訊效能分析報告進行點評。

第三步：學生端操作 (Flipped Workflow)

學生領取作業，獲得個人專屬 Repo，實作 HTTP 請求並錄製網路請求瀑布流圖表後 Push。

第四步：教師回饋 (Review)

進入 Feedback PR，針對學生的「請求瓶頸識別」能力予以回饋，探討如何優化硬體端回應邏輯。
