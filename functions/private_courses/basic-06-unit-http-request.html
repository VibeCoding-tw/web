基礎 6 - 網路通訊基礎

數位世界的溝通橋樑：
HTTP 協議與 Fetch API

從餐廳點餐模型出發，掌握讓網頁、伺服器與無人車互相「對話」的核心技術。

🍽️ 1. 餐廳模型：理解 HTTP 對話 (20 min)

HTTP (HyperText Transfer Protocol) 是網路上交換數據的標準格式。最簡單的理解方式就是「餐廳點餐」：

🙋‍♂️ 請求 (Request)

你向服務生點餐。包含你想看的內容（網址）、你想做的動作、以及你提供的資訊。

🍛 回應 (Response)

服務生送來餐點。包含狀態（成功或賣完）以及具體的內容（網頁、圖檔或 JSON）。

🧠 無狀態性 (Stateless)

伺服器很健忘！每次連線都是獨立的，它不會主動記得你是誰（除非使用 Cookie 或 Token）。

💡 概念解析

為什麼「無狀態性」很重要？這讓伺服器可以同時服務成千上萬的人，而不需要為每個人保留一個龐大的記憶庫。

✉️ 2. 請求的組成：方法、標頭與正文 (40 min)

一封完整的 HTTP 請求信件包含以下關鍵部分：

  • GET

    像看菜單，僅請求數據。

    例如：讀取當下無人車的感測器數值。

  • POST

    像填訂單，發送新數據。

    例如：將無人車路徑紀錄傳回伺服器進行分析。

  • Headers

    信封上的備註。

    告訴伺服器我傳的是 JSON 格式，或者我用的瀏覽器版本。

// URL 參數 (Query Strings) 範例
http://192.168.4.1/move?dir=forward&speed=200

🏷️ 3. 狀態碼：伺服器的表情符號 (20 min)

當伺服器收到請求後，會回傳一個三位數的「狀態碼」來告訴你結果：

✅ 200 OK

請求成功！資料已經準備好送給你了。

🔍 404 Not Found

找不到資源。可能是網址打錯了，或者檔案被刪除了。

💥 500 Error

伺服器內部錯誤。通常是後端的程式碼寫壞了。

🚀 線上任務：手動發送指令

試著在瀏覽器網址列輸入下方網址，並觀察 ESP32 的序列監控視窗接收到的字串：

http://192.168.4.1/move?dir=forward&speed=255

講師手冊 (Instructor Guide)

教學重點 (Teaching Points)

  • 強調 HTTP stateless 的本質：每一次請求都是獨立的。
  • Headers 的角色：解釋為什麼 Content-Type 在發送 JSON 時極為重要。
  • 常用的 Methods：GET (攞) 與 POST (送) 的核心差異。

學生常見問題 (Common Issues)

  • 忘記在 Header 加入 Content-Type: application/json 導致伺服器無法解析。
  • URL 拼寫錯誤（例如少了一個斜線）。
  • 誤以為 GET 請求可以有 Request Body。

評分標準 (Grading)

  • 優 (90-100): 能準確解析不同的狀態碼，並正確處理 JSON 數據發送。
  • 可 (70-89): 基礎連線功能正常，但對於錯誤處理或 Header 設定不夠嚴謹。
  • 待加強 (Below 70): 連線請求失敗，或不理解請求結構的組成。

在 GitHub Classroom 的實戰部署建議

A. 範本倉庫 (Template Repo) 結構

.
├── src/             # 包含 GET/POST 請求封裝的 main.cpp
├── README.md        # 測試報告：Postman 截圖與 ESP32 連線日誌
└── .github/         # 自動評分配置

B. 自動評估 (Autograding) 的設計

  • 核心邏輯檢查： 偵測程式中是否正確應用 http.begin() 以及 http.POST() 函式。
  • 資料格式檢核： 驗證學生提交的 JSON 字串是否符合 RFC 8259 規範，且包含必要的 Key/Value。

GitHub Classroom 詳細操作指南

第一步：準備範本倉庫 (Template Repo)

建立包含基礎 WiFi 連線代碼但未實作 HTTP 請求邏輯的 Repo，於 GitHub 設定中勾選 "Template repository"。

第二步：建立 Classroom 作業

建立 Individual Assignment，連結範本並啟用 "Feedback pull requests" 以便對其 HTTP 封裝邏輯進行點評。

第三步：學生端操作 (Flipped Workflow)

學生領取作業，獲得個人專屬 Repo，實作 GET 與 POST 請求至測試 API 後 Push。

第四步：教師回饋 (Review)

進入 Feedback PR，針對學生的「錯誤狀態碼攔截」與「超時處理」邏輯予以回饋。
