從幾何到代碼：
API 封裝設計

釋放運動控制的潛能。將複雜的 ICC 數學封裝為易用的 API，打造直覺且健壯的控制介面。

📦 1. 為什麼要封裝 API？ (15 min)

好的 API 就像駕駛座的方向盤，駕駛者不需要知道引擎內部的閥門如何開合，只需要轉動方向盤即可。

  • •
    隱藏複雜度 (Abstraction) 使用者只需要呼叫 move(radius, speed)，不需要知道底層的三角函數或 ICC 運算細節。
  • •
    低耦合 (Low Coupling) 若更換了不同輪徑的馬達，只需修改 API 內部的係數，而無需改動上層的邏輯程式碼。

🛡️ 防呆機制 (Fail-safe)

在 API 內部攔截不合理的參數，保護硬體與程式邏輯。

// 例如：防止轉向半徑小於物理極限
if (radius < MIN_RADIUS) {
  radius = MIN_RADIUS;
  log("Warning: Radius clamped");
}

🎛️ 2. 設計良好的控制接口 (20 min)

輸入參數的選取

幾何層級

轉向半徑 ($R$)、中心速度 ($V$)。適合路徑規劃。

時間層級

目標轉向角度 ($\theta$)、角速度 ($\omega$)。適合精確動作。

命名規範 (Naming)

學習使用動詞開頭的命名法，讓程式碼讀起來像是在說故事。

  • ✓ driveArc()
  • ✓ rotateInPlace()
  • ✓ stopGradually()
  • ✗ go() // 太模糊

🏗️ 3. 數據結構與返回類型 (25 min)

API 不應只是執行動作，還應負責資料的打包與狀態回報。

struct WheelSpeeds {
  float left_m_s;
  float right_m_s;
};

狀態回報：API 應回傳執行結果（成功、參數錯誤、硬體異常），而非單純 Void。

📚 學習任務：Arduino Library 規範

查閱 Arduino 官方或第三方函式庫設計，思考為什麼大多數都使用 begin()、set() 與 get() 前綴？

關鍵概念：抽象化層級

設計一個 move(linear, angular) 函式，而不是傳統且依賴硬體的 setMotor(pwmL, pwmR)。

作業指引 (Assignment Guide)

作業 2-1：封裝運動大腦：Kinematics 類別設計

  • 角色扮演： 軟體架構師。
  • 執行方式： 建立一個 `DifferentialDrive` 類別。實作 `calculateIK(radius, speed)` 函式，並封裝 `L` (輪距) 等硬體常數。
  • 評分重點： 代碼的結構化程度與可讀性。是否正確處理了 \(R\) 為 `Infinity`（直線走）的情境。

作業 2-2：API 調用挑戰：畫出 8 字形

  • 角色扮演： 測試工程師。
  • 執行方式： 禁止直接寫 PWM 數值，僅能透過呼叫自定義 API 組合出「右轉半圓」與「左轉半圓」的指令。
  • 評分重點： 代碼的簡潔度，以及 8 字形軌跡的對稱性與回到原點的精準度。

作業 2-3：擴充 API：目標角度轉向控制 (turnAngle)

  • 角色扮演： 韌體工程師.
  • 執行方式： 結合 `delay()` 或非阻塞計時器，實作 `turnAngle(degrees, speed)`。讓車子能精準轉向 90 度後停下。
  • 評分重點： 時間與角速度的轉換邏輯是否正確，以及在 README 中記錄轉向的累積誤差。
