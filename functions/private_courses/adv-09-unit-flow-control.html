前端資料緩衝與流控
Debouncing & Throttling

優化數據傳輸頻率，消除硬體延遲與不必要的運算，讓操控體驗更絲滑。

🚧 1. 為什麼需要流控？ (20 min)

在即時物聯網應用中，「發送越多越好」往往是錯誤的觀念。未經節流的數據流會導致嚴重的系統瓶頸。

硬體瓶頸

ESP32 的藍牙接收與緩衝區有上限。如果網頁發送太快（例如 60Hz），封包會開始排隊 (Queuing)，導致你放開搖桿後車子還在跑，這就是所謂的「操作延遲感」。

無謂的運算

並非所有微小的座標變動都需要傳送到馬達。過濾掉手抖產生的「雜訊」，能顯著節省電力與無線頻寬。

⚔️ 2. 兩大技術武器 (30 min)

💡 防抖 (Debouncing)

像聲控燈。

你一直叫它就不滅，直到你「安靜了一段時間」它才執行。

  • 適合：輸入框搜尋 (Search Input)
  • 適合：設定變更確認

🚇 節流 (Throttling)

像捷運閘門。

管你外面排多少人，每 100ms 只放行一個。保證執行頻率不超過上限。

  • 適合：搖桿移動 (Joystick)
  • 適合：視窗縮放 (Resize)

🧪 學習任務

觀察視覺化演示工具，理解「執行最後一次 (Last exec)」與「固定頻率執行 (Fixed rate)」的邏輯差異。

🧠 3. 高階流控與優化 (20 min)

Threshold (變動閥值)

儲存上一次發送的數值 \( V_{last} \)，只有當 \( |V_{current} - V_{last}| > Threshold \) 時才執行傳輸。

有效過濾手指的生理顫抖，或 AI 辨識出的重複手勢指令。

Deadzone (死區)

當搖桿位移在中心點 \(\pm 10\%\) 範圍內時，強制輸出為 0。防止馬達因為微小訊號而發出低頻嗡嗡聲。

// 變動閥值邏輯示意
let lastVal = 0;
const THRESHOLD = 5;

function update(currentVal) {
  if (Math.abs(currentVal - lastVal) > THRESHOLD) {
    sendToBLE(currentVal);
    lastVal = currentVal;
  }
}

流控實驗室：打造絲滑操控

透過代碼與參數調校，在靈敏度與效能之間找到黃金平衡點。

⏱️ 60 min

【作業 4-1】搖桿節流器

任務內容：在 onMove 事件中限制發送頻率。

  • 計時器限制：利用 setTimeout 確保頻率不超過 20Hz (50ms)。
  • 效能觀測：使用 Chrome DevTools 比較加入前後的封包量。
🎯 60 min

【作業 4-2】數據過濾

任務內容：實作數值過濾器，減少無效傳輸。

  • Deadzone：中心 \(\pm 10\%\) 強制歸零。
  • 變動觸發：數值變化 \(> Threshold\) 才發送。
📉 45 min

【作業 4-3】壓力測試

任務內容：模擬極限情況並找出最佳參數。

  • 延遲測量：記錄這「按下」到「ESP32收到」的時間。
  • 參數優化：測試 10ms - 500ms 節流區間的手感變化。
