進階 7 - 單元 4

事件驅動 vs 輪詢
Event-Driven Architecture

優化數據存取效能：理解中斷 (Interrupt) 與輪詢 (Polling) 的差異，打造高效的即時監控系統。

🍕 1. 觀念拆解：外送員與餐廳的溝通 (25 min)

  • 輪詢 (Polling)

    像是一個飢餓的顧客，每隔 1 分鐘就打電話問餐廳「我的飯好了嗎？」。

    ✅ 邏輯簡單、可預測執行時間 ❌ 浪費資源、高手延遲

  • 事件驅動 (Event-Driven)

    像是顧客點完餐後去做自己的事，等餐廳主動「傳簡訊」通知領餐。

    ✅ 省電、零閒置浪費、即時反應 ❌ 程式流程跳躍、初學者難追蹤

🔄 2. 瀏覽器中的事件與循環 (25 min)

// 事件驅動 vs 輪詢實作範例
// 事件驅動
btn.addEventListener('click', () => { ... });
// 輪詢更新
setInterval(() => {
if (checkStatus()) { handleResult(); }
}, 100);
  • addEventListener：理解瀏覽器如何監聽藍牙斷線、按鈕點擊。
  • setInterval/setTimeout：理解如何利用定時器實作輪詢。

🔍 學習任務

思考：為什麼「搖桿控制」適合輪詢，而「藍牙連線成功」適合事件驅動？

🚨 3. 硬體端的中斷 (Interrupts) (10 min)

ESP32 attachInterrupt：理解當網頁傳送「緊急停止」或車子撞到東西時，硬體如何立即跳出主程式進行處理（事件驅動的極致表現）。

🧠 進階思考

比較 WebSocket.onmessage (事件驅動) 與 setInterval + Fetch (輪詢) 的優缺點。何時該追求「極低延遲」與「省電穩定」？

進階課程 7 - 單元 4 | 事件驅動 vs. 輪詢更新實務

作業指引 (Assignment Guide)

作業 4-1：效率大挑戰：從輪詢到監聽

  • 角色扮演： 介面效能優化師。
  • 執行方式： 實作一個 WiFi 密碼輸入框。先使用 `setInterval` 每 100ms 檢查一次內容長度（輪詢），再改用 `input` 事件監聽器（事件驅動）實作一樣的功能。
  • 評分重點： 能否說明「事件驅動」在輸入框場景下為什麼能節省不必要的 CPU 運算。

作業 4-2：「手感微調」：搖桿的自適應輪詢

  • 角色扮演： 遙控系統開發者。
  • 執行方式： 實作一個「定時發送器」。將搖桿 `mousemove` 事件的高頻座標存入變數，但限制每隔 50ms 才從該變數撈出數值並透過藍牙發出一次。
  • 評分重點： 此種「混合模式 (Hybrid)」是否能有效解決數據發送過快導致的 BLE 連線崩潰問題，並在畫面上顯示數據流入與發出的頻率對比。

作業 4-3：硬體緊急停止 (Fail-safe) 實作

  • 角色扮演： 安全系統工程師。
  • 執行方式： 模擬「中斷 (Interrupt)」的概念。利用藍牙的 `gattserverdisconnected` 事件，一旦連線中斷，立即彈出全畫面警告並清空所有馬達指令。
  • 評分重點： 系統對於「突發事件」的反饋及時性，對比「定時詢問是否斷線」所造成的延遲差異。
