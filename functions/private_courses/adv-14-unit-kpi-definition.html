系統性能指標 (KPI) 定義

沒有指標就沒有真相。學習如何量化、監控並優化無人車系統的真實表現。

為什麼需要指標？

量化優化：當你修改了 $K_p$ 值或縮小了影像解析度，效能提升了 $5\%$ 還是 $50\%$？工程師如果不相信直覺，只相信數據。
SLA (服務水準協議)：在專業領域中，系統必須承諾最低標準（例如：控制延遲必須小於 $100ms$）。

核心概念

  • ✓KPI 三大維度：時間、準確性、可靠性
  • ✓權衡 (Trade-off)：FPS 與 Latency 的拉鋸
  • ✓數據驅動 (Data-driven) 的開發流程

1 關鍵性能指標 (KPI) 三大維度

⏱️時間性 (Temporal)

  • 延遲 (Latency)：從攝影機拍到畫面，經過處理，到馬達真正轉動的總時間。
  • 幀率 (FPS)：每秒能夠處理並作出決策的影像數量。

🎯準確性 (Accuracy)

  • 平均誤差 (Mean Error)：質心偏離影像中心線的平均像素值。
  • 震盪指數 (Jitter)：控制輸出訊號的波動頻率與振幅，反映行駛的平穩度。

🛡️可靠性 (Reliability)

  • 任務完成率：成功跑完一圈 S 彎道且未出界的次數比例。
  • 斷線復歸時間：BLE 意外斷線後，系統重新連線並恢復控制所需的時間。

2 權衡與組合指標

在資源有限的嵌入式系統中，我們往往無法兼得所有優點。

思考題：哪一個對「避障」更有利？

  • 方案 A：高 FPS (60Hz) 但 高延遲 (500ms) —— 看得快但反應慢
  • 方案 B：低 FPS (10Hz) 但 低延遲 (50ms) —— 看得慢但反應快

對於高速移動的無人車，低延遲通常比高 FPS 更為關鍵。為了綜合評估，我們會定義「複合指標 (Score)」，例如：
Performance Score = (FPS * 0.4) + (1000/Latency * 0.6)

官方性能規格書範例：
在 $320 \times 240$ 解析度下，穩定維持 $20 FPS$ 且延遲 $< 80ms$。這就是你的系統必須達成的 SLA。

📝 實作任務 (Classroom Challenges)

Task 2-1 「數位儀表板」實作 (60 min)

任務說明 (Mission)

讓抽象的指標變成具體的可視化資訊，在 Web UI 上實作即時性能監控面板。

  • 即時計算：在 JavaScript 中使用 performance.now() 計算每輪控制循環的精確耗時。
  • 滾動平均：計算最近 10 幀的平均延遲，避免數據跳動過劇烈。
  • 視覺化：在網頁上用紅/黃/綠燈號標示狀態（例如：< 50ms 為綠燈，> 150ms 為紅燈）。

Task 2-2 「壓力極限測試」實驗 (60 min)

任務說明 (Mission)

透過人為增加負載，找出系統崩潰的臨界點，並學習硬體基準測試 (Benchmarking)。

  • 變因測試：逐步增加影像解析度 (QVGA -> VGA)，觀察 FPS 下降到多少時，車子開始無法穩定循跡。
  • 誤差紀錄：紀錄在不同車速下，過彎時的「最大瞬間誤差值」。
  • 產出：為你的無人車寫下「官方性能規格書」。

Task 2-3 「KPI 優化賽」 (45 min)

任務說明 (Mission)

練習數據驅動開發 (Data-driven Development)，針對某一項 KPI 進行針對性優化。

  • 選擇目標：選擇「降低延遲」或「提高精準度」作為單一目標。
  • 實驗對照：修改代碼（如優化 JSON 打包或調整濾波演算法），並與優化前的數據對比。
  • 效益分析：計算並提交你的「優化率 %」。
