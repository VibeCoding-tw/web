數據、速度與控制：
數據流與取樣率

理解機器人學的隱藏法則：如何在高頻率更新與通訊瓶頸之間找到完美的「甜蜜點」。

📡 1. 什麼是取樣率？ (Sampling Rate)

核心概念：取樣率是指每秒鐘讀取感測器或發送控制指令的次數，單位是 Hz (赫茲)。

🚶

1 Hz

每秒更新一次

「車子像喝醉一樣，無法修正路線。」

🚗

10 Hz

每 0.1 秒更新一次

「基本的遙控手感，能開但反應稍慢。」

🛸

50-100 Hz

專業級更新率

「流暢手感，專業無人機或平衡車的標準。」

⚠️ 2. 數據流的瓶頸：頻寬與延遲

為什麼我們不乾脆設到 1000Hz？因為硬體有其極限：

📦 藍牙 MTU (最大傳輸單元)

藍牙每次傳輸的封包大小有限制（通常是 20 到 200 字節）。如果數據太長，會被拆成多個封包，增加延遲。

🧠 處理時間 (Parsing Time)

ESP32 解析一個複雜的 JSON 封包需要時間。如果發送頻率高過解析速度，系統就會卡死 (Lag)。

🌀 3. 奈奎斯特與混疊 (Aliasing)

理論基礎：奈奎斯特定理

「若要精確捕捉某種動作，取樣頻率必須大於該動作頻率的兩倍。」(fs > 2f)

🎥 生活中的現象：倒轉的電風扇

當電風扇轉很快時，在相機畫面裡看起來會像在倒著轉？這就是 混疊 (Aliasing)。因為相機的取樣率（幀率）不足以追上扇葉的速度，導致大腦誤判了運動方向。

📦 4. 數據封裝與調度

定時發送 vs. 有數據就發送

「有數據就送」在搖桿快速亂動時會瞬間產生大量封包，導致藍牙擁塞。使用 節流 (Throttling) 技術，確保最高取樣率被限制在 50Hz，系統會更穩定。

🛠️ 線上任務：JSON 設計

設計一個精簡的 JSON 格式，將以下數據打包在一起回傳：

  • 電池電壓 (V)
  • 左/右馬達轉速 (RPM)
  • 超音波距離 (cm)
// 精簡後的 JSON 範例
{
  "v": 7.42,
  "rpm": [120, 118],
  "d": 15.5
}

💡 提示：使用短 Key 名稱 (如 v 代替 batteryVoltage) 可以大幅減少字節數，提高取樣率容量。

作業指引 (Assignment Guide)

【作業 3-1】「抖動測試」：頻率對手感的影響

  • 角色扮演： 操縱感調校工程師 (Control Feel Tuning Engineer)。
  • 執行方式： 在前端 App 實作滑桿，讓發送頻率可動態從 1Hz 調整至 100Hz。實施通過障礙物的側向移動測試，對比 5Hz 與 50Hz 的駕駛手感差異。
  • 評分重點 : 成功實作頻率切換邏輯，並能清楚描述高取樣率如何減少控制時的「頓挫感」。

【作業 3-2】感測器數據降噪：滑動平均濾波

  • 角色扮演： 數據科學家 (Data Scientist)。
  • 執行方式： 建立一個長度為 10 的陣列作為 Buffer。對 50Hz 高頻取樣下的超音波數據進行「先進先出」式平滑處理。對比處理前後的距離跳動圖表。
  • 評分重點 : 濾波器邏輯正確，數據輸出明顯比原始採樣更平滑。

【作業 3-3】事件驅動 vs. 定時取樣 (節流機制)

  • 角色扮演： 通訊協定優化專家 (Protocol Optimization Expert)。
  • 執行方式： 模擬快速亂動搖桿觸發的高頻封包。實作 Throttling 機制，確保無論輸入頻率多快，藍牙發送頻率最高限制在 50Hz (每 20ms 一次)。
  • 評分重點 : 節流邏輯能有效防止藍牙 MTU 擁塞導致的 Lag 或是 ESP32 系統當機。

C. 自動評分 (Autograding) 的設計

  • 邏輯檢測：： 驗證遞迴或循環緩衝區 (Circular Buffer) 的數據更新指標計算是否會發生溢位。
  • 通訊開銷估計：： 根據學生設計的 JSON 長度與頻率，自動計算並警示是否超過 BLE 常規頻寬。
