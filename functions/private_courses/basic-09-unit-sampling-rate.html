數據、速度與控制：
數據流與取樣率

理解機器人學的隱藏法則：如何在高頻率更新與通訊瓶頸之間找到完美的「甜蜜點」。

📡 1. 什麼是取樣率？ (Sampling Rate)

核心概念：取樣率是指每秒鐘讀取感測器或發送控制指令的次數，單位是 Hz (赫茲)。

🚶

1 Hz

每秒更新一次

「車子像喝醉一樣，無法修正路線。」

🚗

10 Hz

每 0.1 秒更新一次

「基本的遙控手感，能開但反應稍慢。」

🛸

50-100 Hz

專業級更新率

「流暢手感，專業無人機或平衡車的標準。」

⚠️ 2. 數據流的瓶頸：頻寬與延遲

為什麼我們不乾脆設到 1000Hz？因為硬體有其極限：

📦 藍牙 MTU (最大傳輸單元)

藍牙每次傳輸的封包大小有限制（通常是 20 到 200 字節）。如果數據太長，會被拆成多個封包，增加延遲。

🧠 處理時間 (Parsing Time)

ESP32 解析一個複雜的 JSON 封包需要時間。如果發送頻率高過解析速度，系統就會卡死 (Lag)。

🌀 3. 奈奎斯特與混疊 (Aliasing)

理論基礎：奈奎斯特定理

「若要精確捕捉某種動作，取樣頻率必須大於該動作頻率的兩倍。」(fs > 2f)

🎥 生活中的現象：倒轉的電風扇

當電風扇轉很快時，在相機畫面裡看起來會像在倒著轉？這就是 混疊 (Aliasing)。因為相機的取樣率（幀率）不足以追上扇葉的速度，導致大腦誤判了運動方向。

🤔 線上思考任務：

如果你的輪胎轉速是每秒 20 圈 (20Hz)，但你的光電編碼器取樣率只有 30Hz，會發生什麼事？

📦 4. 數據封裝與調度

定時發送 vs. 有數據就發送

「有數據就送」在搖桿快速亂動時會瞬間產生大量封包，導致藍牙擁塞。使用 節流 (Throttling) 技術，確保最高取樣率被限制在 50Hz，系統會更穩定。

🛠️ 線上任務：JSON 設計

設計一個精簡的 JSON 格式，將以下數據打包在一起回傳：

  • 電池電壓 (V)
  • 左/右馬達轉速 (RPM)
  • 超音波距離 (cm)
// 精簡後的 JSON 範例
{
  "v": 7.42,
  "rpm": [120, 118],
  "d": 15.5
}

💡 提示：使用短 Key 名稱 (如 v 代替 batteryVoltage) 可以大幅減少字節數，提高取樣率容量。

實戰挑戰：優化數據流

學會管理數據，讓控制變得精準且優雅。

🕹️ 60 min

【作業 3-1】

「抖動測試」

在前端實作滑桿，動態調整發送頻率 (1Hz - 100Hz)。

  • 實機駕駛：挑戰在 5Hz 與 50Hz 下通過障礙物。
  • 觀察：100Hz 時是否出現 Lag 或斷連？
  • 目標：找出流暢與穩定的「甜蜜點」。
📈 60 min

【作業 3-2】

「滑動平均濾波」

處理 50Hz 高頻取樣下的超音波跳動：

  • 建立長度為 10 的陣列作為 Buffer。
  • 實現「先進先出」數據更新邏輯。
  • 體驗：感受「平滑度提升」與「時間延遲增加」之間的權衡。
🛡️ 45 min

【作業 3-3】

「強韌通訊機制」

比較「搖桿一動就傳」與「20ms 定時傳」：

  • 擁塞模擬：快速亂動搖桿，觀察系統是否卡死。
  • 節流實作：確保無論手速多快，傳輸頻率最高僅 50Hz。

講師手冊 (Instructor Guide)

教學重點 (Teaching Points)

  • 頻率的權衡：為什麼不是採樣越快越好（CPU 負擔、電力消耗、無線傳輸頻寬）。
  • 奈奎斯特準則：解釋採樣頻率必須大於訊號最高頻率的兩倍，否則會失真。
  • 即時性 (Real-time)：探討採樣頻率對無人車反應速度的直接影響。

學生常見問題 (Common Issues)

  • 採樣頻率設定太高，導致序列埠 (Serial) 輸出阻塞，反而拖慢了整個 loop。
  • 忘記轉換單位：混淆了 Hz (次/秒) 與 ms (毫秒/次) 的計算。
  • 不理解為什麼慢速採樣會導致感測器數據噴發「鋸齒狀」的雜訊。

評分標準 (Grading)

  • 優 (90-100): 能準確設計多個頻率層級的採樣任務，且理解其對系統性能的量化影響。
  • 可 (70-89): 基本功能達成，但對採樣頻率的選擇缺乏理論支持（如：隨機設定數字）。
  • 待加強 (Below 70): 無法穩定控制採樣頻率，導致數據流混亂。
