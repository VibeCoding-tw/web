基礎 7 - 網路通訊架構

ESP32 Web Server 入門：
從「阻塞」到「非同步」的進化

擺脫 loop() 的束縛，實作基於事件驅動的高性能網頁伺服器。

🛡️ 1. 認識 ESP32 的網頁守門員 (20 min)

核心概念：區分傳統的阻塞式與現代的非同步處理方式。

🐌 舊式 WebServer.h (Sync)

  • • 必須在 loop() 呼叫 handleClient()。
  • • 如果 loop 忙碌（如影像處理），網頁會沒反應。
  • • 就像一個只能一次點餐一位的櫃檯。

⚡ 新式 ESPAsyncWebServer (Async)

  • • 基於事件驅動，背後自動處理請求。
  • • 不佔用 loop() 時間，最適合即時控制。
  • • 就像百貨公司美食街，點完餐領取號碼牌（Callback）。

🧐 學習任務

觀看動畫短片，思考：為什麼在控制「高速移動」的無人車時，非同步處理比傳統處理更安全？

🗺️ 2. 路由 (Routing) 與 處理器 (Handlers) (20 min)

核心概念：將特定的 URL 路徑應對到 C++ 的執行內容。

// 匿名函式 (Lambda) 語法範例
server.on("/led/on", HTTP_GET, [](AsyncWebServerRequest *request) {
  digitalWrite(LED_PIN, HIGH);
  request->send(200, "text/plain", "LED is ON");
});

💡 關鍵技術：Lambda 語法

[] (AsyncWebServerRequest *request) { ... } 是 C++11 的匿名函式。它讓我們在宣告路由的當下，就定義好處理邏輯，讓程式碼更簡潔。

⚙️ 學習任務

查閱函式庫文件，理解 request->send() 除了傳送純文字，還可以傳送哪些類期的數據？（例如：HTML、JSON 或 SPIFFS 檔案）

🎨 3. 靜態資源管理 (15 min)

關鍵問題：將整個 HTML 寫在 C++ 字串裡有什麼缺點？（維護困難、縮排亂掉）。

🖌️ 解決方案：R"rawliteral(...)"

使用 C++ 的原始字串字面量，可以保留 HTML 的換行與雙引號，不需要每次都加反斜線轉義。

const char index_html[] PROGMEM = R"rawliteral(
<!DOCTYPE html>
<html>
  <h1>My Controller</h1>
</html>
)rawliteral";

🚀 線上實作任務

在 CodePen 或線上 HTML 編輯器中撰寫一段簡單的控制頁面原型（包含兩個按鈕：前進、後退）。完成後，思考如何將這段內容嵌入 ESP32。

講師手冊 (Instructor Guide)

教學重點 (Teaching Points)

  • 非阻塞的概念：為什麼使用 ESPAsyncWebServer 不會拖慢 loop() 的執行速度。
  • 路由 (Routing) 邏輯：如何將不同的 URL 路徑對應到特定的功能塊。
  • HTTP 參數處理：如何從 GET 請求的 URL 中提取數值。

學生常見問題 (Common Issues)

  • 在非同步回呼函式中直接呼叫耗時操作（如 delay），這會破壞非同步的優勢。
  • 忘記定義 404 Not Found 處理，導致輸入錯誤網址時伺服器無響應。
  • 不理解指標 (Pointer) 在 on() 函式中的參數傳遞。

評分標準 (Grading)

  • 優 (90-100): 能高效地組織路由代碼，正確處裡各類請求參數，且伺服器表現極致流暢。
  • 可 (70-89): 基本伺服器功能正常，但代碼重用性較低，或對非同步請求的底層原理不夠熟悉。
  • 待加強 (Below 70): 無法成功架設 Web 伺服器，或在處理請求時造成 ESP32 系統死機。

B. 自動評估 (Autograding) 的設計

  • 路由有效性檢測： 透過網路腳本發送 HTTP 請求（如 /status, /toggle），驗證是否能獲取 200 OK 響應。
  • 非同步安全性檢查： 掃描是否存在 `delay` 關鍵字，確保學生正確應用非同步編程模型。
