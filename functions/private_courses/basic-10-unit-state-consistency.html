基礎 10 - 單元 3

非同步通訊
當你的螢幕說謊時

在指令發送與硬體執行之間，如何確保「網頁狀態」與「機器人現實」永遠同步？

🏁 1. 競爭條件 (Race Condition) (30 min)

核心概念：在非同步環境下，多個操作同時發生，最終結果取決於誰「跑得快」（抵達順序）。

場景模擬：

  • 網頁發送「切換至自動模式」。
  • 同一時間，車子偵測到電力不足，自動切回「待機模式」。
  • 如果「待機回傳」比「模式確認」早到，網頁可能會卡在錯誤的狀態。

🎯 學習目標

理解 發送順序 $\neq$ 抵達順序。在不穩定的網路中，舊的數據可能會蓋掉新的數據。

✨ 2. 樂觀 UI (Optimistic UI) 的代價 (20 min)

定義：點擊按鈕後立即改變介面顏色（假設成功），而不是等硬體回傳。這在社交媒體（如點讚）很常見，但在硬體控制中很危險。

✅ 優點

極致的操作流暢感，使用者不會感覺到毫秒級的延遲。

❌ 缺點

如果硬體端切換失敗，介面必須「閃回」原狀，造成使用者困惑甚至引發危險。

🔄 3. 定時輪詢 vs. 事件推送 (20 min)

❓ 輪詢 (Polling)

網頁主動、定時地詢問：「你現在是什麼模式？」。優點是實作簡單，缺點是浪費頻寬且有同步延遲。

🔔 事件推送 (Push / Notify)

硬體狀態一改變，主動通知網頁。優點是效率極高、低延遲，缺點是實作較複雜（需建立雙向隧道）。

💡 預習任務：思考題

「在藍牙 (BLE) 或 WiFi 頻寬有限且網路不穩定的情況下，哪種方式較能確保顯示器與無人車的『真實狀態』一致？」

提示：考慮丟包 (Packet Loss) 的情況。

核心任務：解決「同步焦慮」

實作專業的同步策略，確保遠端操作的安全與精確。

🔢 60 min

【作業 3-1】

「版本序號」同步法

  • 任務：為每一個狀態變更賦予一個遞增的 Sequence Number。
  • 核心邏輯：網頁收到封包時，檢查序號是否大於上一次。
  • 挑戰：模擬網路波動，觀察系統是否能自動忽略「遲到的舊資訊」。
⚖️ 60 min

【作業 3-2】

應答確認機制 (ACK/NACK)

  • 任務：實作具備「超時重試」功能的發送函式。
  • 視覺回饋：在確認收到 ACK 前，按鈕顯示「轉圈圈」動畫。
  • 目標：消除使用者對於指令是否傳遞成功的焦慮。
🏠 45 min

【作業 3-3】

真值來源 (SSoT) 重構

  • 任務：強制 UI 遵循「單向數據流」，嚴禁樂觀更新。
  • 測試：對比於延遲下，兩者的操作「體感差異」。
  • 目標：理解安全性優先場景下，「精確」重於「反應」。

講師手冊 (Instructor Guide)

教學重點 (Teaching Points)

  • 非同步的代價：解釋為什麼「發送順序」不等於「抵達順序」。
  • 樂觀更新 (Optimistic UI) 的適用場景：什麼時候可以使用？（如點讚）什麼時候禁止？（如啟動馬達）。
  • 單一真值來源 (SSoT)：強調分散式控制系統中，實體硬體的狀態才是最終真理。

學生常見問題 (Common Issues)

  • 不理解為什麼介面會「跳動」：這是因為舊的狀態回傳蓋掉了新的操作指令。
  • ACK 機制寫得太複雜，導致網路擁塞（解釋指數退避或簡單超時計數）。
  • 混淆了 TCP 的穩定性與應用層的狀態一致性。

評分標準 (Grading)

  • 優 (90-100): 能完美解決 Race Condition，UI 在極端延遲下仍能展現正確且穩定的狀態。
  • 可 (70-89): 具備基本的同步意識，但在邏輯邊界情況（如極短時間內連續連擊）可能失效。
  • 待加強 (Below 70): UI 與硬體狀態長期不一致，且無任何錯誤過濾機制。

在 GitHub Classroom 的實戰部署建議

A. 範本倉庫 (Template Repo) 結構

.
├── src/             # 包含網路延遲模擬與 ACK 邏輯的 main.cpp
├── README.md        # 測試報告：在 500ms 延遲下的 UI 穩定性截圖
└── .github/         # 自動評估系統

B. 自動評估 (Autograding) 的設計

  • 數據一致性檢查： 模擬亂序封包，檢查學生代碼是否能透過 Sequence Number 過濾舊數據。
  • 通訊效率評核： 檢查 ACK 回傳頻率，避免無意義的頻繁應答導致網路擁塞。

GitHub Classroom 詳細操作指南

第一步：準備範本倉庫 (Template Repo)

建立包含 SSoT (單一真值來源) 框架的 Repo，於 GitHub 設定中勾選 "Template repository"。

第二步：建立 Classroom 作業

建立 Individual Assignment，連結範本並啟用 "Feedback pull requests" 以進行細部代碼通訊邏輯點評。

第三步：學生端操作 (Flipped Workflow)

學生點擊作業連結，獲得個人專屬 Repo，Clone 到本地開發並 Push 回傳觀察其對「非同步衝突」的處理能力。

第四步：教師回饋 (Review)

進入 Feedback PR，針對學生的 Sequence Number 與 ACK 機制實作方案給予回饋。
