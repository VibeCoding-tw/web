基礎 10 - 單元 3

非同步通訊
當你的螢幕說謊時

在指令發送與硬體執行之間，如何確保「網頁狀態」與「機器人現實」永遠同步？

🏁 1. 競爭條件 (Race Condition) (30 min)

核心概念：在非同步環境下，多個操作同時發生，最終結果取決於誰「跑得快」（抵達順序）。

場景模擬：

  • 網頁發送「切換至自動模式」。
  • 同一時間，車子偵測到電力不足，自動切回「待機模式」。
  • 如果「待機回傳」比「模式確認」早到，網頁可能會卡在錯誤的狀態。

🎯 學習目標

理解 發送順序 $\neq$ 抵達順序。在不穩定的網路中，舊的數據可能會蓋掉新的數據。

✨ 2. 樂觀 UI (Optimistic UI) 的代價 (20 min)

定義：點擊按鈕後立即改變介面顏色（假設成功），而不是等硬體回傳。這在社交媒體（如點讚）很常見，但在硬體控制中很危險。

✅ 優點

極致的操作流暢感，使用者不會感覺到毫秒級的延遲。

❌ 缺點

如果硬體端切換失敗，介面必須「閃回」原狀，造成使用者困惑甚至引發危險。

🔄 3. 定時輪詢 vs. 事件推送 (20 min)

❓ 輪詢 (Polling)

網頁主動、定時地詢問：「你現在是什麼模式？」。優點是實作簡單，缺點是浪費頻寬且有同步延遲。

🔔 事件推送 (Push / Notify)

硬體狀態一改變，主動通知網頁。優點是效率極高、低延遲，缺點是實作較複雜（需建立雙向隧道）。

💡 預習任務：思考題

「在藍牙 (BLE) 或 WiFi 頻寬有限且網路不穩定的情況下，哪種方式較能確保顯示器與無人車的『真實狀態』一致？」

提示：考慮丟包 (Packet Loss) 的情況。

核心任務：解決「同步焦慮」

實作專業的同步策略，確保遠端操作的安全與精確。

🔢 60 min

【作業 3-1】

「版本序號」同步法

  • 任務：為每一個狀態變更賦予一個遞增的 Sequence Number。
  • 核心邏輯：網頁收到封包時，檢查序號是否大於上一次。
  • 挑戰：模擬網路波動，觀察系統是否能自動忽略「遲到的舊資訊」。
⚖️ 60 min

【作業 3-2】

應答確認機制 (ACK/NACK)

  • 任務：實作具備「超時重試」功能的發送函式。
  • 視覺回饋：在確認收到 ACK 前，按鈕顯示「轉圈圈」動畫。
  • 目標：消除使用者對於指令是否傳遞成功的焦慮。
🏠 45 min

【作業 3-3】

真值來源 (SSoT) 重構

  • 任務：強制 UI 遵循「單向數據流」，嚴禁樂觀更新。
  • 測試：對比於延遲下，兩者的操作「體感差異」。
  • 目標：理解安全性優先場景下，「精確」重於「反應」。

作業指引 (Assignment Guide)

【作業 3-1】實作「版本序號」同步法

  • 角色扮演： 分散式系統工程師 (Distributed Systems Engineer)。
  • 執行方式： 為狀態變更封包加上遞增的 Sequence Number。在網頁端實作邏輯：僅接受序號大於目前存儲值的資料包。模擬網路延遲環境，測試系統是否能自動過濾「遲到」的過時訊息。
  • 評分重點 : 成功偵測並忽略舊序號封包，UI 狀態不再因延遲而產生「時光倒流」的閃爍現象。

【作業 3-2】實作應答確認機制 (ACK/NACK)

  • 角色扮演： 通訊協定設計師 (Communication Protocol Designer)。
  • 執行方式： 實作具備「超時重試」與「狀態回執」功能的發送框架。在網頁按鈕點擊後顯示「讀取中」動畫，直到收到硬體回傳的 ACK (確認) 或 NACK (拒絕) 為止。
  • 評分重點 : 成功處理指令丟包情況，具備自動重試邏輯，且 UI 能清晰反饋指令的執行結果。

【作業 3-3】真值來源 (SSoT) 重構

  • 角色扮演： 系統架構師 (System Architect)。
  • 執行方式： 強制 UI 模式遵循「單向數據流」：點擊按鈕不立即改變狀態，而是等待硬體端的主動推送（通知）。對比「樂觀更新」與「SSoT 控制」在不穩定的無線通訊下的穩定表現。
  • 評分重點 : 徹底移除 UI 端的所有「先行假設」，確保網頁顯示與機器人物理現狀 100% 一致。

GitHub Classroom 詳細操作指南

第一步：準備範本倉庫 (Template Repo)

  1. 在 GitHub 建立新 Repo（名為 basic-10-sync-template）。
  2. 上傳包含藍牙通訊框架但「未處理數據競爭 (Racing)」的實驗程式。
  3. 在 Repo 設定中勾選 "Template repository"。

第二步：建立 Classroom 作業

  1. 進入 GitHub Classroom 建立「Individual Assignment」。
  2. 連結上述的範本倉庫，並勾選 "Enable feedback pull requests"。

第三步：學生端操作 (Flipped Workflow)

  1. 學生領取作業，獲得個人專屬 Repo。
  2. 模擬長延遲 (Long Latency) 環境，實作版本序號過濾器與 ACK 重傳邏輯。
  3. 完成 SSoT 結構重構後 Push 代碼 提交進度。

第四步：教師回饋 (Review)

  1. 進入 Feedback PR 的 Files Changed。
  2. 重點評價學生對「安全（精確性）高於體驗（反應速度）」這一權衡的理解程度。

在 GitHub Classroom 的實戰部署建議

此單元聚焦於「分散式系統的一致性」，引導學生理解在不穩定的無線環境下，如何建立強韌的雙向通訊協議。

A. 範本倉庫 (Template Repo) 結構

.
├── src/
│   └── main.cpp      # (重點) 需實作 Sequence Check 邏輯
├── tests/            # 模擬封包丟失與亂序的測試腳本
└── README.md         # 實驗心得：版本序號如何解決 UI 閃跳問題

B. 參考範例代碼 (Reference Example Code)

main.cpp (Sequence Filter)

static uint32_t lastSeq = 0;
void onData(Data d) {
  if (d.seq > lastSeq) {
    lastSeq = d.seq;
    updateUI(d.state);
  }
}

C. 自動評分 (Autograding) 的設計

  • 一致性檢測：： 透過 Python 測試腳本連續發送亂序封包，驗證 UI 最終狀態是否停留在最新序號的資料。
  • ACK 健全度：： 斷開一部分通訊，檢查學生程式是否會觸發自動重傳 (Exponential Backoff 為優)。
